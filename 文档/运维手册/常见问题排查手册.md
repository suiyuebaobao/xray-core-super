# 常见问题排查手册

本文档提供 suiyue 系统的常见故障排查和应急处理指南。

## 1. 服务启动失败

### 1.1 API 容器启动后立即退出

检查日志：

```bash
docker-compose logs api
```

常见原因：

| 症状 | 原因 | 解决 |
|------|------|------|
| `JWT secret is too weak` | JWT_SECRET 使用默认值 | 修改 `.env` 中的 `JWT_SECRET` 为强随机值 |
| `dial tcp 127.0.0.1:3306` | 数据库连接失败 | 检查 MySQL 容器是否运行，`DATABASE_URL` 是否正确 |
| `table doesn't exist` | 未执行迁移 | `make migrate` |

### 1.2 MySQL 容器启动失败

```bash
docker-compose logs mysql
```

- 数据目录权限问题：`sudo chown -R 999:999 /var/lib/docker/volumes/suiyue_mysqldata`
- 端口冲突：MySQL 不暴露到宿主机，无需检查端口冲突

## 2. 认证问题

### 2.1 用户登录失败

1. 检查用户名密码是否正确
2. 查看 API 日志确认 bcrypt 验证过程
3. 检查用户状态是否为 `active`

```sql
SELECT id, username, status FROM users WHERE username = 'admin';
```

### 2.2 Token 刷新失败

- 检查浏览器是否发送了 `refresh_token` Cookie（HttpOnly Cookie 需 HTTPS）
- 开发环境通过 `localhost` 访问时 Cookie 正常工作
- 检查 `refresh_tokens` 表中是否存在对应记录

```sql
SELECT * FROM refresh_tokens WHERE user_id = 1 ORDER BY expires_at DESC LIMIT 5;
```

### 2.3 管理员权限拒绝（403）

确认用户 `is_admin` 字段为 `true`：

```sql
UPDATE users SET is_admin = true WHERE username = 'admin';
```

## 3. 订阅问题

### 3.1 订阅下载返回 404

- 检查订阅 token 是否存在于 `subscription_tokens` 表
- 检查对应订阅状态是否为 `ACTIVE`
- 检查是否过期（`expire_date` < 当前时间）
- 检查流量是否超限：
  普通流量看 `used_traffic >= traffic_limit`
  家宽流量看 `residential_used_traffic >= residential_traffic_limit`
  `used_traffic` 与 `residential_used_traffic` 是按套餐倍率后的扣费流量；真实节点增量看 `usage_ledgers.delta_total`，扣费增量看 `usage_ledgers.billed_total`。
- 检查套餐是否关联了节点组，且节点组通过 `node_group_nodes` 绑定了启用节点

```sql
SELECT st.token, us.status, us.expire_date,
       us.used_traffic, us.traffic_limit,
       us.residential_used_traffic, us.residential_traffic_limit
FROM subscription_tokens st
JOIN user_subscriptions us ON us.user_id = st.user_id
WHERE st.token = 'xxx';
```

### 3.2 订阅内容为空（无节点）

确认：
1. 套餐关联了节点组：`SELECT * FROM plan_node_groups WHERE plan_id = ?`
2. 节点组下有启用节点：

```sql
SELECT n.*
FROM node_group_nodes ngn
JOIN nodes n ON n.id = ngn.node_id
WHERE ngn.node_group_id = ? AND n.is_enabled = true;
```

### 3.3 订阅可导入但节点无信号

优先检查订阅下发参数是否和节点 Xray 实际配置一致，Reality 的 `server_name`/`public_key` 任意一个不一致都会握手失败。

```sql
SELECT id, name, host, port, server_name, public_key, short_id, fingerprint, flow
FROM nodes
WHERE is_enabled = true;
```

在节点服务器核对：

```bash
cat /usr/local/etc/xray/config.json
xray run -test -config /usr/local/etc/xray/config.json
systemctl status xray node-agent
```

需要重点确认：
- `nodes.server_name` 等于 Xray `realitySettings.serverNames[0]`
- `nodes.public_key` 等于 Xray `privateKey` 派生出的 PublicKey，或配置中的 `realitySettings.publicKey`
- 当前用户 UUID 已存在于 Xray `settings.clients`
- Xray 配置包含可用 `outbounds`，例如 `freedom`
- `nodes.traffic_pool` 是否和该线路预期一致；若订阅里少的是家宽节点，优先检查家宽流量是否耗尽，而不是普通流量

如果订阅接口返回 `40007 当前套餐没有可用节点`，优先检查该用户当前套餐是否绑定了节点分组、节点分组里是否有启用出口节点，以及该流量池是否仍有剩余额度。新建 UAT 套餐后必须先绑定包含真实出口节点的节点分组，再导入客户端测试。

如果用户兑换了包含家宽流量的套餐，但用户订阅页家宽流量仍为 0，检查是否命中了旧版本兑换逻辑遗留数据：

```sql
SELECT us.id, u.username, us.plan_id, us.residential_traffic_limit, p.residential_traffic_limit AS plan_residential
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
JOIN plans p ON p.id = us.plan_id
WHERE us.status = 'ACTIVE'
  AND us.residential_traffic_limit = 0
  AND p.residential_traffic_limit > 0;
```

新版本兑换码续费会同时叠加普通流量和家宽流量，且保留两个池的已用量。

### 3.4 中转线路可导入但不可用

中转线路排查时要分清入口和出口：

1. 订阅 URI 的 `server/port` 应该是中转节点 `relays.host:relay_backends.listen_port`
2. 订阅 URI 的 `sni/pbk/sid` 应该是出口节点 Reality 参数
3. 中转节点监听端口必须 TCP 可达
4. 中转节点到出口节点 `target_host:target_port` 必须 TCP 可达
5. 出口节点防火墙必须允许中转节点 IP 访问 Xray 端口
6. 用户 UUID 必须存在于出口节点 Xray `clients`
7. node-agent relay 模式使用的 `RELAY_TOKEN` 只能访问中转任务接口，不能当作用户订阅 Token 使用
8. HAProxy 配置必须校验通过并成功 reload

示例检查：

```bash
# 用户侧或中心侧检查中转入口
nc -vz relay-hk.example.com 24443

# 中转节点上检查出口连通
nc -vz hk.example.com 443

# 中转节点上检查 HAProxy
haproxy -c -f /etc/haproxy/haproxy.cfg
systemctl status haproxy

# 出口节点上查看防火墙策略
iptables -S
ufw status
```

不要把多个 Reality 参数不同的出口节点挂在同一个中转监听端口后面做负载均衡，否则客户端可能随机握手失败。

## 4. 节点通信问题

### 4.0 一键部署不稳定或新服务器失败

一键部署应走 Docker 容器闭环，不应依赖人工 SSH 修改节点配置。后台部署日志需要逐步确认：

1. 镜像预检成功，API 容器能读取 `/root/raypilot-artifacts/node-agent-image.tar.gz`
2. 服务器预检成功，目标机 `/tmp` 空间充足且具备 `curl`、`sh`、包管理器等基础命令
3. Docker 安装或 `docker info` 成功
4. 容器处于 Running，且日志没有 `xray-core setup failed`、`haproxy setup failed`
5. 出口节点已读到 `/usr/local/etc/xray/config.json` 并写回 Reality 参数
6. 出口节点或 node_host 有新心跳，中转 relay 有新心跳
7. 出口节点 `nodes.port` 已由 `xray` 监听，或中转 `listen_port` 已由 `haproxy` 监听；如果端口被 nginx、宝塔或其他进程占用，必须先释放端口后再部署
8. 公网 TCP 端口可达；一键部署会自动放行 UFW/firewalld，但云安全组或上游 ACL 仍需单独放行

常用命令：

```bash
docker compose exec api ls -lh /root/raypilot-artifacts/node-agent-image.tar.gz
docker info
docker ps -a --filter name=raypilot-node-agent
docker ps -a --filter name=raypilot-relay-agent
docker logs --tail 160 raypilot-node-agent
docker logs --tail 160 raypilot-relay-agent
ss -lntp
ufw status
nc -vz <node-ip> 443
nc -vz <node-ip> 8443
cat /usr/local/etc/xray/config.json
```

如果目标机是脏环境，部署流程会删除旧 node-agent 容器、旧 systemd 服务、旧 Xray/HAProxy 配置并重新生成。失败后应先看部署日志中的“容器诊断”，不要手工改配置后把半成品当成功。

XHTTP 常见症状：TCP `443` 节点正常，但 XHTTP `8443` 节点无信号或客户端日志显示 `dial tcp <ip>:8443: i/o timeout`。优先确认：

```bash
ss -lntp | grep 8443
ufw status | grep 8443
nc -vz <node-ip> 8443
```

如果 `ss` 显示 `xray` 已监听但公网 `nc` 超时，说明不是订阅格式或 Reality 参数问题，而是防火墙、安全组或上游 ACL 未放行该端口。

### 4.1 节点心跳失败

- 检查 `nodes` 表中的 `agent_base_url` 和 `agent_token_hash` 是否正确
- 确认 node-agent 进程正常运行
- 检查网络连通性：`curl http://node-ip:port/health`

### 4.2 任务未执行

检查任务表：

```sql
SELECT id, action, status, retry_count, last_error, node_id
FROM node_access_tasks
WHERE status IN ('PENDING', 'PROCESSING', 'FAILED')
ORDER BY created_at DESC
LIMIT 20;
```

- 状态为 `PENDING`：等待节点心跳拉取
- 状态为 `PROCESSING`：节点已领取，等待执行完成
- 状态为 `FAILED`：超过重试次数，需人工处理

### 4.3 任务堆积处理

对于长期失败的任务，可以重置重试计数：

```sql
UPDATE node_access_tasks SET status = 'PENDING', retry_count = 0, locked_at = NULL, lock_token = NULL
WHERE status = 'FAILED' AND retry_count < 10;
```

## 5. 流量统计问题

### 5.1 流量未正确统计

- 检查 node-agent 是否定期上报流量数据
- 检查 `traffic_snapshots` 表是否有增量记录
- 确认快照差值法正常工作：当前快照 >= 上次快照
- 如果节点心跳正常但 `traffic_snapshots`、`usage_ledgers` 长期为 0，优先检查出口节点 node-agent 是否已更新到启用 Xray Stats API 的版本
- 节点侧执行：

```bash
xray api statsquery --server 127.0.0.1:10085 -pattern 'user>>>' -reset=false
```

若返回连接失败或空错误，检查 `/usr/local/etc/xray/config.json` 是否包含 `api`、`stats`、`policy.levels.0.statsUserUplink/statsUserDownlink` 和 tag 为 `api` 的本地入站。新版 node-agent 启动时会自动补齐旧配置；一键部署出口节点会传入 `XRAY_API_SERVER=127.0.0.1:10085`。

### 5.1.1 多出口 IP 节点订阅可见但出站不正确

- 确认目标服务器只运行一个当前角色的 node-agent，旧 systemd `node-agent`、旧 node-agent 二进制和旧 relay agent 容器都已清理。
- 检查中心 `nodes.listen_ip`、`nodes.outbound_ip` 与节点 `/usr/local/etc/xray/config.json` 中对应 inbound `listen`、outbound `sendThrough` 是否一致。
- 确认多出口逻辑节点已经加入套餐使用的节点组；只部署成功但未加入节点组时，订阅不会下发该节点，也不会触发已有活跃订阅的用户同步。
- 检查 `node_access_tasks` 中新逻辑节点的 UPSERT_USER 任务是否为 DONE；节点 Xray `clients` 必须包含用户 UUID。
- 使用 Xray 客户端或本地 socks 代理分别连接每条多出口线路，访问 `https://api.ipify.org`，返回值必须等于该逻辑节点的 `outbound_ip`。
- 如果旧中转角色已被清理，必须同步停用 `relays` 和 `relay_backends`，否则订阅仍可能下发不可用中转线路。

### 5.1.2 家宽 SOCKS5 节点源 IP 与最终出口不一致

- `nodes.outbound_ip` 在 `outbound_type=socks5` 时表示连接上游 SOCKS5 的本机源 IP，不表示上游代理最终家宽 IP。
- 检查节点 Xray 配置中该逻辑节点的 `socks` outbound 是否包含 `sendThrough`，且值等于中心 `nodes.outbound_ip`。
- 客户端访问 `https://api.ipify.org` 看到的应该是上游 SOCKS5 返回的家宽出口 IP；如果要确认本机源 IP 是否生效，需要在上游代理服务商侧查看连接来源，或在节点服务器上用 `curl --interface <outbound_ip> -x socks5h://... https://api.ipify.org` 对比。
- 家宽 SOCKS5 节点必须清空 `flow`。如果订阅或 Xray client 里仍有 `xtls-rprx-vision`，即使 TCP 传输也可能导致线路无信号；应检查 `nodes.outbound_type=socks5` 的 `flow` 是否为空、节点同步任务 payload 是否不含 `flow`，以及远端 Xray clients 是否已重同步。
- 如果家宽经节点转发后访问 Cloudflare 明显卡顿、慢或触发风控，优先检查 Clash/mihomo 订阅中该线路是否为 `udp: false`。中心字段是 `nodes.udp_enabled`，SOCKS5 家宽默认应为 false；这会让客户端避免 QUIC/HTTP3 走二次转发，通常比继续开 UDP 更稳定。确有游戏、语音等 UDP 业务时再按线路开启。

### 5.2 流量重置（节点重启）

当 `new < old` 时，判定为计数器重置，差值按 0 处理，重建基线。这是正常行为。

### 5.3 node-agent 中心地址不可达

当前中心服务器主/备入口通过未提交的环境变量维护：

```bash
主入口: <primary-center-url>
备用入口: <fallback-center-url-a>
备用入口: <fallback-center-url-b>
```

节点上的 agent 应同时配置：

```bash
CENTER_SERVER_URL=<primary-center-url>
CENTER_SERVER_URLS=<primary-center-url>,<fallback-center-url-a>,<fallback-center-url-b>
```

排查命令：

```bash
# Docker agent
docker inspect raypilot-node-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL
docker inspect raypilot-relay-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL
docker logs --tail 100 raypilot-node-agent
docker logs --tail 100 raypilot-relay-agent

# systemd agent
systemctl cat node-agent
journalctl -u node-agent --no-pager -n 100

# 节点侧连通性
curl -i <fallback-center-url-a>/health
curl -i <fallback-center-url-b>/health
```

### 5.4 控制台页面正常但 API 502

如果前端页面能打开，但登录、套餐、节点等 `/api/*` 请求返回 502，优先检查 Nginx 是否缓存了旧 API 容器 IP。Compose 滚动替换 API 容器后，旧容器 IP 会失效，Nginx 配置必须通过 Docker DNS 动态解析 `api`。

```bash
docker compose exec nginx getent hosts api
docker compose exec nginx nginx -t
docker compose exec nginx nginx -s reload
curl -i http://127.0.0.1/api/plans
```

`deploy/nginx/default.conf` 应包含：

```nginx
resolver 127.0.0.11 valid=10s ipv6=off;
set $api_upstream http://api:3000;
proxy_pass $api_upstream;
```

处理方式：

- 首选后台“修复中心”：出口节点在 `/admin/nodes` 操作，中转节点在 `/admin/relays` 操作。
- 修复接口会通过 SSH 重建 Docker agent 或更新 systemd drop-in，并可等待新心跳。
- 部署日志类型为 `agent_center_repair`，可在 `/admin/logs` 的部署日志中查看步骤和失败原因。

## 6. 数据库操作

### 6.1 紧急禁/启用户

```sql
-- 禁用用户
UPDATE users SET status = 'disabled' WHERE id = ?;

-- 启用用户
UPDATE users SET status = 'active' WHERE id = ?;
```

禁用用户后，Worker 会在下次扫描时创建 `DISABLE_USER` 任务并下发到节点。

### 6.2 手动创建订阅

```sql
INSERT INTO user_subscriptions (user_id, plan_id, start_date, expire_date, traffic_limit, used_traffic, status)
VALUES (1, 1, NOW(), DATE_ADD(NOW(), INTERVAL 30 DAY), 10737418240, 0, 'ACTIVE');

-- 用户级订阅 token 可提前创建；subscription_id 可为空
INSERT INTO subscription_tokens (user_id, subscription_id, token, is_revoked)
VALUES (1, NULL, 'your-high-entropy-token', false);
```

### 6.3 数据库备份与恢复

备份：

```bash
docker-compose exec mysql mysqldump -u root -p suiyue > backup_$(date +%Y%m%d_%H%M%S).sql
```

恢复：

```bash
docker-compose exec -T mysql mysql -u root -p suiyue < backup_20260425.sql
```

### 6.4 数据库表检查

```sql
SHOW TABLES;
SELECT COUNT(*) FROM users;
SELECT COUNT(*) FROM user_subscriptions;
SELECT COUNT(*) FROM nodes WHERE is_enabled = true;
```

## 7. Worker 问题

### 7.1 Worker 不工作

```bash
docker-compose logs worker
docker-compose restart worker
```

Worker 负责以下定时任务：
- 扫描过期订阅 → 创建 `DISABLE_USER` 任务
- 清理过期 Refresh Token
- 重试失败任务

### 7.2 手动触发扫描

进入 Worker 容器：

```bash
docker-compose exec worker /app/worker
```

Worker 基于 robfig/cron 调度，启动后自动执行定时任务。
