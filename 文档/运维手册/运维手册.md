# 运维手册

本文档提供 suiyue（岁月）系统的日常运维、故障排查和应急响应指南。

## 系统架构概览

```
用户/客户端
    │
    ▼
Nginx (80/443) ← 静态资源 + API 反代
    │
    ├── /api/*  → 后端 API (:3000)
    ├── /sub/*  → 后端 API (:3000)
    └── /*      → 前端静态文件
                    │
                    ▼
               MySQL (容器内部，不暴露端口)

节点服务器
    │
    ├── xray-core (443) ← VLESS + Reality
    └── node-agent (systemd 服务)
              │
              ▼
         心跳 → 中心 API
         任务 ← 中心 API
         流量 → 中心 API

中转服务器
    │
    ├── HAProxy ← TCP 透传转发
    └── node-agent (AGENT_ROLE=relay)
              │
              ▼
         心跳 → 中心 API
         配置任务 ← 中心 API
         结果 → 中心 API
```

## 日常运维

### 服务启动/停止

```bash
# 启动所有服务
make up

# 停止所有服务
make down

# 仅启动 API 服务
make api

# 启动 Worker
make worker

# 启动前端开发服务器
make frontend
```

### 日志查看

```bash
# 查看所有服务日志
docker-compose logs -f

# 查看 API 日志
docker-compose logs -f api

# 查看 Worker 日志
docker-compose logs -f worker

# 查看 MySQL 日志
docker-compose logs -f mysql

# 查看 node-agent 日志（节点服务器）
journalctl -u node-agent -f

# 查看 xray-core 日志（节点服务器）
journalctl -u xray -f
```

日志中心 v1 口径：

- 后台 `运行日志` 页面读取宿主机 `logs/api.log` 与 `logs/worker.log`
- Docker Compose 部署会把宿主机 `./logs` 挂载到 API/Worker 容器 `/app/logs`，API 标准输出同时写入 `api.log`，Worker 标准输出同时写入 `worker.log`
- 新环境首次启动时日志文件尚未产生，运行日志接口返回空列表，不应返回 500；若页面无日志，先确认 `logs/` 目录权限和 Compose 挂载
- 后台 `部署日志` 页面展示结构化的一键部署记录、步骤明细和失败原因
- 后台 `操作日志` 页面展示用户关键行为和管理员关键操作
- 结构化日志必须记录 `client_ip`，并保留 `X-Forwarded-For` / `X-Real-IP` 原始值用于反代链路排障
- 部署日志还必须记录 `operator_ip`、`target_server_ip`、`selected_exit_ips`、`relay_entry_ip`、`target_host`，出口节点部署只保存 `outbound_type`、`outbound_ip` 和 `outbound_proxy_count` 摘要，不保存完整 SOCKS5 账号密码
- 中心地址 SSH 兜底修复会记录为 `agent_center_repair` 部署日志，保存目标服务器、节点/物理主机/中转 ID、中心地址列表和步骤结果；不得保存 SSH 密码、完整 Token 或私钥
- 日志中不得写入密码、完整 Token、数据库连接串和 Reality 私钥

### Nginx 反代检查

Compose 内置 Nginx 通过 Docker DNS `127.0.0.11` 动态解析 `api` 服务名。滚动更新 API/Worker 后不需要停止项目，但要确认 Nginx 配置仍包含 `resolver 127.0.0.11` 与变量形式的 `proxy_pass`，避免 Nginx 缓存旧 API 容器 IP 后 `/api/*` 返回 502。

```bash
docker compose exec nginx nginx -t
docker compose exec nginx nginx -s reload
curl -i http://127.0.0.1/api/plans
```

## 生产前验收清单

生产发布前必须把“功能可用”和“风险边界”分开记录。当前核心路径通过 smoke 不代表零缺陷；以下检查全部通过后，才可以认为本次版本具备上线条件。

### 基础构建与测试

```bash
go test ./...
docker-compose config --services
cd frontend
npm run build
E2E_ADMIN_USERNAME=<admin> E2E_ADMIN_PASSWORD=<password> npm run test:e2e -- --workers=1
```

Playwright 默认访问 `http://127.0.0.1:7000`，公网验收时使用：

```bash
cd frontend
E2E_BASE_URL=http://your-domain-or-ip E2E_ADMIN_USERNAME=<admin> E2E_ADMIN_PASSWORD=<password> npm run test:e2e -- --workers=1
```

多出口 IP 节点需要把真实 multi_exit 流量纳入验收时，额外设置：

```bash
E2E_REQUIRE_MULTI_TRAFFIC=1 \
E2E_MULTI_NODE_HOST_ID=<node-host-id> \
E2E_MULTI_TRAFFIC_NODE_IDS=<node-id-a,node-id-b> \
E2E_MULTI_NODE_HOST_TOKEN=<node-host-token> \
E2E_ADMIN_USERNAME=<admin> \
E2E_ADMIN_PASSWORD=<password> \
npm run test:e2e -- --workers=1
```

### 订阅与节点检查

- 订阅 URL 必须返回 200，且客户端导入的是完整链接 `/sub/{token}/clash`、`/sub/{token}/base64` 或 `/sub/{token}/plain`，不是裸 Token。
- Reality 节点必须确认订阅里的 `sni/pbk/sid` 与出口节点 Xray 配置一致。
- 出口节点必须确认用户 UUID 已写入 Xray `clients`，并通过真实客户端或 Xray 本地代理完成一次出站访问。
- 多出口 IP 节点必须确认目标服务器只有一个当前角色的 node-agent，每个逻辑节点的 Xray inbound `listen` 与 outbound `sendThrough` 都绑定同一个公网 IP，订阅下发的每条线路真实出站 IP 等于对应逻辑节点 IP。
- 中转线路必须确认中转入口端口可达、中转到出口端口可达、`relay_config_tasks` 最新任务为成功状态、`relay_traffic_snapshots` 有流量记录。
- 已清理或停用的旧中转节点必须同步停用 `relays` 与 `relay_backends`；停用出口或改作管理系统的服务器也必须停用相关 `relay_backends` 并触发一次中转 `RELOAD_CONFIG`，确认 HAProxy 不再转发到旧目标。
- 如果要隐藏出口 IP，必须把出口节点设置为只下发中转线路，并在出口节点防火墙只放行中转服务器 IP 访问 Xray 端口。

### 尚未覆盖的风险

- 长时间稳定性观察和并发压测。
- 支付回调验签、幂等、对账和异常补偿全链路。
- 管理后台主路径新增、编辑、删除、禁用、恢复已由 Playwright 覆盖；复杂组合、长时间稳定性和并发压测仍需按生产发布单独验收。
- 多中转节点、多出口节点、大量用户同步任务下的 Worker 重试表现。

## 故障排查

### 控制平台迁移或中心地址不可达

当前控制平台有一个主域名和两个公网备用入口：

```bash
主入口: http://leiyunai.fun
备用入口: http://154.219.106.105
备用入口: http://154.219.106.53
```

一键部署出口节点和中转节点时，agent 容器必须同时具备：

```bash
CENTER_SERVER_URL=http://leiyunai.fun
CENTER_SERVER_URLS=http://leiyunai.fun,http://154.219.106.105,http://154.219.106.53
```

排查顺序：

```bash
# 1. 从节点服务器验证主备中心都能访问
curl -i http://154.219.106.105/health
curl -i http://154.219.106.53/health

# 2. 检查 Docker agent 的环境变量
docker inspect raypilot-node-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL
docker inspect raypilot-relay-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL

# 3. 检查 systemd agent 的 drop-in
systemctl cat node-agent
systemctl cat raypilot-node-agent
```

如果节点无法自动切换中心，在后台执行：

- 出口节点：`/admin/nodes` -> 对应节点服务器 -> “修复中心”。
- 中转节点：`/admin/relays` -> 对应中转 -> “修复中心”。

修复入口会通过 SSH 登录节点服务器，重建 Docker agent 或写入 systemd drop-in，并可等待 `node_id`、`node_host_id` 或 `relay_id` 新心跳。等待超时表示中心地址已写入但还未确认新心跳，需要查看远端 agent 日志。不要在未知状态下手工删除运行中的 agent 容器；Docker 修复脚本会在新容器启动失败时恢复旧容器。

### 用户反馈订阅失效

```sql
-- 检查用户订阅状态
SELECT us.id, us.user_id, us.status, us.expire_date, u.username
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE u.username = 'username';

-- 检查 Token 是否有效
SELECT token, user_id, subscription_id, is_revoked, expires_at, last_used_at
FROM subscription_tokens
WHERE token = 'token_string';

-- 检查 xray 配置中是否有该用户
-- 登录节点服务器执行：
cat /usr/local/etc/xray/config.json | grep "user_key"
```

如果订阅能导入但节点无信号，优先核对 Reality 参数：

```sql
SELECT id, name, host, port, server_name, public_key, short_id
FROM nodes
WHERE is_enabled = 1;
```

节点侧执行：

```bash
cat /usr/local/etc/xray/config.json
xray run -test -config /usr/local/etc/xray/config.json
systemctl status xray node-agent
```

`nodes.server_name` 必须等于 `realitySettings.serverNames[0]`，`nodes.public_key` 必须等于节点 PublicKey，`nodes.short_id` 必须等于 `shortIds[0]`。

如果是中转线路，继续检查：

```bash
# 检查中转入口端口
nc -vz <RELAY_HOST> <LISTEN_PORT>

# 登录中转服务器，检查到出口节点的连通性
nc -vz <EXIT_NODE_HOST> 443

# 登录出口节点，确认只放行中转 IP 时防火墙规则生效
iptables -S
ufw status
```

中转线路的 `server/port` 来自中转节点，但 `sni/pbk/sid` 必须仍然来自出口节点。隐藏出口 IP 时，还必须在出口节点防火墙限制只允许中转服务器访问 Xray 端口。

### 节点离线排查

### 流量统计异常

```sql
-- 检查用户当前订阅使用量（普通流量 + 家宽流量）
SELECT us.id, u.username,
       us.traffic_limit, us.used_traffic,
       ROUND(us.used_traffic * 100.0 / NULLIF(us.traffic_limit, 0), 2) AS normal_percent_used,
       us.residential_traffic_limit, us.residential_used_traffic,
       ROUND(us.residential_used_traffic * 100.0 / NULLIF(us.residential_traffic_limit, 0), 2) AS residential_percent_used
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE us.status = 'ACTIVE';

-- 检查最近流量快照
SELECT node_id, xray_user_key, uplink_total, downlink_total, captured_at
FROM traffic_snapshots
ORDER BY captured_at DESC
LIMIT 10;

-- 检查最近账本流量池归属
SELECT node_id, traffic_pool, xray_user_key, delta_total, recorded_at
FROM usage_ledgers
ORDER BY recorded_at DESC
LIMIT 20;
```

排查时必须额外确认：

- `nodes.traffic_pool` 是否符合该节点预期，普通节点应为 `normal`，家宽节点应为 `residential`
- 普通流量异常时看 `used_traffic`，家宽流量异常时看 `residential_used_traffic`
- 单个流量池耗尽后，只应影响该池节点和对应中转线路，不应导致另一池节点一起消失

## 应急响应

### 数据库密码泄露

```bash
# 1. 修改 MySQL root 密码
docker-compose exec mysql mysql -u root -p -e "ALTER USER 'root'@'%' IDENTIFIED BY 'new_password'; FLUSH PRIVILEGES;"

# 2. 更新 DATABASE_URL 环境变量
# 编辑 .env 文件，更新 DATABASE_URL

# 3. 重启服务
docker-compose restart
```

### JWT 密钥泄露

```bash
# 1. 生成新的 JWT_SECRET
openssl rand -base64 64

# 2. 更新 .env 文件
# JWT_SECRET="new-secret-key"

# 3. 重启 API 服务（所有现有 Token 失效）
docker-compose restart api
```

### 恶意节点/Token

```sql
-- 撤销特定订阅 Token
UPDATE subscription_tokens SET is_revoked = 1 WHERE token = 'malicious_token';

-- 管理员重置用户 Token 时应通过后台接口完成，避免手工 SQL 生成低熵 Token。

-- 禁用特定用户
UPDATE users SET status = 'disabled' WHERE username = 'malicious_user';

-- 禁用特定节点
UPDATE nodes SET is_enabled = 0 WHERE id = <NODE_ID>;
```

## 监控指标

### 关键指标

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| API 响应时间 | > 1s | 可能数据库慢查询或负载过高 |
| MySQL 连接数 | > 80% | 可能连接泄漏 |
| node-agent 心跳间隔 | > 60s | 节点可能离线 |
| node-agent relay 模式心跳间隔 | > 60s | 中转配置可能未生效 |
| 中转监听端口可达率 | < 100% | 用户中转线路可能不可用 |
| HAProxy 配置校验失败 | > 0 | 中转配置未 reload，需检查 relay_config_tasks 错误 |
| 磁盘使用率 | > 80% | 日志或数据库文件过大 |
| 内存使用率 | > 90% | 可能内存泄漏 |
