# 代理订阅系统 - 前后端分离开发方案 v3.3

## 1. 目标

构建一套面向 `xray-core` 节点的订阅分发和用户管理系统。节点数据面统一使用 `VLESS + Reality`，节点控制面统一通过 `node-agent` 管理，覆盖以下核心能力：

- 用户注册、登录、鉴权
- 用户中心页面和后台管理页面
- 套餐管理和订阅开通
- 订阅链接分发，支持 Clash/mihomo、Base64 聚合、纯文本 URI 三种格式
- 节点分组和套餐授权
- 直连线路与中转线路并存，一个用户订阅 Token 可下发多条线路
- 多出口 IP 服务器可由管理员手动开启多 IP 模式，一台物理服务器一个 node-agent，多个公网出口 IP 拆成多个逻辑出口节点
- 流量统计、配额控制、到期控制
- 兑换码开通订阅
- USDT 支付订单和回调开通（v1 保留骨架，不上线）
- 前端使用 Vue 3 + Element Plus 实现前后端分离
- Docker 化部署

非目标：

- 不做前端商城复杂装修
- 不做多币种支付网关聚合
- 不做通用面板，先服务于当前自有节点体系

## 2. 核心设计原则

### 2.1 协议与运行时约束

节点运行时统一使用 `xray-core`，入站协议统一以 `VLESS + Reality` 为主，不再基于 `vmess + alterId` 设计订阅模板。

订阅输出支持三种格式：
- `Clash/mihomo` YAML：适用于 Clash Verge Rev、mihomo 类客户端
- `Base64` 聚合订阅：适用于大多数通用客户端（Shadowrocket、Surge、Quantumult 等）
- 纯文本 URI 链接：最原始分享格式，兼容性最广

### 2.2 节点控制面约束

每台出口节点部署两个核心组件：

- `xray-core`：负责真实代理转发，属于数据面
- `node-agent`：负责本机 `xray-core` 的用户增删改、启停、状态采集、流量上报，属于控制面

每台中转节点部署两个核心组件：

- HAProxy：第一版默认 TCP 转发组件，负责中转入口端口到出口节点端口的 TCP 透传
- `node-agent` relay 模式：负责中转节点心跳、转发配置任务拉取、配置渲染、reload 和结果上报

node-agent 使用同一个工程和二进制，通过运行角色区分职责：

- `AGENT_ROLE=exit`：出口节点模式，使用 `NODE_ID` + `NODE_TOKEN`，绑定 `nodes` 表，管理 xray 用户和流量
- `AGENT_ROLE=relay`：中转节点模式，使用 `RELAY_ID` + `RELAY_TOKEN`，绑定 `relays` 表，只管理中转端口和转发配置

node-agent 出口模式具备自动部署能力：
- 启动时自动检测本机是否已安装 xray-core
- 未安装时自动下载并安装最新版 xray-core
- 自动生成 VLESS + Reality 默认配置文件（含 x25519 密钥对）
- 自动注册并启动 xray systemd 服务
- 零人工干预即可完成节点初始化

服务端职责：

- 对出口节点只和 `node-agent` exit 模式通信；对中转节点只和 `node-agent` relay 模式通信
- 不直接跨公网调用每台机器上的 `xray-core gRPC`
- 订阅授权、开通、禁用、超额封禁都通过 `agent` 落地到节点

这样可以把中心服务和节点运行时解耦，避免把节点本地实现细节暴露给服务端。中转能力复用 node-agent 的部署、鉴权、心跳和任务框架，但出口用户同步任务与中转配置任务必须分表、分 Token、分运行角色。

### 2.3 直连与中转分层约束

系统必须同时支持两种数据面线路：

- **直连线路**：客户端直接连接出口节点 `nodes.host:nodes.port`，Reality 握手终点是出口节点本机 `xray-core`。
- **中转线路**：客户端连接中转节点 `relays.host:relay_backends.listen_port`，中转节点只做 TCP 透传，再转发到某个出口节点 `nodes.host:target_port`。Reality 握手终点仍然是出口节点。

中转能力不替代现有直连能力。现有 `nodes` 在业务语义上统一称为**出口节点**，继续由 `node-agent` exit 模式管理用户 UUID、流量上报和 Xray Reality 参数。新增的 `relays` 是**中转入口服务器**，由 `node-agent` relay 模式管理监听端口和转发目标，不写入用户 UUID，不做订阅鉴权，不做用户级流量计费。

订阅侧保持一个用户一个订阅 Token：

- 一个 Token 内可以同时包含直连线路和中转线路
- 直连线路的 `server/port` 来自出口节点
- 中转线路的 `server/port` 来自中转节点和监听端口
- 两类线路的 `uuid/sni/pbk/sid/fingerprint/flow/transport/xhttp-*` 都必须来自同一个出口节点；XHTTP 线路不下发 `flow`

用户连接中转时不需要额外的“中转用户 Token”。客户端发起 VLESS + Reality 连接时携带用户 UUID，中转节点只做 TCP 透传，出口节点 xray 最终识别该 UUID 并完成用户鉴权和套餐扣量。

系统内 Token 必须分清三类：

- 用户订阅 Token：给客户端下载订阅使用，一个用户一个
- 出口节点 `NODE_TOKEN`：给 `node-agent` exit 模式访问中心服务使用，绑定 `nodes.id`
- 中转节点 `RELAY_TOKEN`：给 `node-agent` relay 模式访问中心服务使用，绑定 `relays.id`

隐藏出口 IP 的正确做法是：

1. 订阅只下发中转线路，不下发直连线路
2. 出口节点防火墙只放行中转服务器 IP 访问 Xray 端口
3. 中转节点仅做 TCP passthrough，不终止 TLS/Reality

第一版中转授权不引入新的线路组模型，沿用现有“套餐 -> 节点分组 -> 出口节点”授权。只要套餐允许访问某个出口节点，订阅即可下发该出口节点的直连线路，以及所有指向该出口节点且启用的中转线路。后续需要更细粒度时，再引入 `lines`、`line_groups`、`plan_line_groups`。

一个中转监听端口默认只能绑定一个出口节点。除非多个出口节点共享完全一致的 Reality 身份（同一组 `server_name/public_key/short_id/privateKey`），否则不能在同一个中转端口后面做多出口负载均衡，否则 Reality 握手会随机失败。

多出口 IP 服务器不默认扫描。管理员在一键部署前必须显式选择“多 IP 服务器”，系统才通过 SSH 扫描服务器公网 IPv4，并用 `curl --interface <IP>` 验证实际出口 IP。只有管理员手动勾选确认的可用公网 IP 才会创建为 `nodes` 逻辑出口节点。未开启多 IP 模式时，一键部署必须保持单节点行为，不得自动创建多个节点。

### 2.4 套餐与基础套餐约束

系统必须始终存在一个基础套餐，`plans.is_default=true` 表示该套餐。基础套餐不能删除，只能修改，且后台会强制保持启用。

用户注册和管理员新增用户时，系统自动创建基础套餐订阅并生成用户唯一订阅 Token。删除普通套餐时不硬删历史订单和兑换码，而是将套餐标记为 `is_deleted=true` 并从套餐列表隐藏；仍在使用该套餐的活动订阅会自动迁移到基础套餐，重置为基础套餐的流量和时长，并触发出口节点用户同步。没有活动订阅的用户也会自动补齐基础套餐订阅。

后台套餐管理页面必须直接展示“基础套餐”标识，并禁用基础套餐删除按钮。节点分组页面选择的是出口节点，界面需要明确展示出口节点的 `line_mode`，让管理员能直接判断该节点会下发直连线路、中转线路或两者同时下发。

### 2.5 订阅分发方式

客户端应直接访问一个可下载的 HTTPS 链接，而不是先调 JWT JSON 接口再自己解包。

推荐形式：
- `GET /sub/:token/clash` — 返回 Clash YAML
- `GET /sub/:token/base64` — 返回 Base64 聚合订阅
- `GET /sub/:token/plain` — 返回纯文本 URI 链接

返回内容直接对应格式，不做 JSON 包装。这样更符合客户端实际接入方式，也便于后续做缓存、限流和失效控制。

### 2.6 流量统计方式

流量统计必须按"快照差值"计算，不能把累计计数器每次直接入库后再求和，否则会重复计费。

当前约束下，服务端应接收节点 `agent` 上报的累计计数；`agent` 在节点本地通过 Xray Stats API 读取 `user>>>xray_user_key>>>traffic>>>uplink/downlink`，但这层实现细节不暴露给中心服务。出口模式默认使用 `XRAY_API_SERVER=127.0.0.1:10085`，node-agent 启动时必须补齐 Xray 配置中的 `api`、`stats`、用户级 `policy` 和本地 API 入站。

正确做法：

- 周期性从 agent 上报获取每个用户在每个节点上的累计 uplink/downlink
- 保存最近一次快照
- 本次快照减上次快照，得到增量
- 只把增量累加到用户订阅使用量

### 2.7 支付闭环

支付记录必须先和订单绑定，再由订单驱动订阅开通。

正确链路：

1. 用户创建订单
2. 订单绑定用户、套餐、金额、收款地址、状态
3. 支付回调命中订单
4. 校验金额、地址、交易哈希、确认数
5. 在事务中更新订单、写支付记录、开通订阅

v1 阶段保留订单和支付相关表结构及代码骨架，暂不上线实际支付功能。用户开通订阅主要通过兑换码或管理员后台手动开通。

### 2.8 v1 冻结业务规则

为避免开工后反复返工，`v1` 业务规则在本方案内先冻结为以下版本：

- 订阅 token 允许多端共用，v1 不做设备数限制
- 续费只做时长叠加：若存在 ACTIVE 订阅，则在当前到期时间基础上延长，不重置已用流量
- 超流量后立即禁用节点访问，并使订阅下载接口返回不可用状态
- 每个用户分配一个全局稳定的 xray_user_key，v1 不按订阅轮换
- node-agent 与中心服务采用全量主动推送模式，agent 定时上报状态并拉取待执行任务
- node-agent 鉴权采用独立 token，v1 不上 mTLS

### 2.9 前后端分离

v1 采用前后端分离架构：

- 后端：Go + Gin 提供 REST API 和订阅下载接口
- 前端：Vue 3 + Vite + Element Plus 独立构建，单项目通过路由区分用户端和管理端
- Nginx 统一反向代理，前端静态资源和后端 API 通过同域名不同路径区分
- 前端不再与服务端模板混合，首屏渲染由 Vue SPA 负责

### 4.1 前后端分离实现方式

- `cmd/api` 启动同一个 Gin 应用，提供 REST API 和订阅下载接口
- `cmd/worker` 只负责后台定时任务，不提供页面
- `frontend/` 是独立的 Vue 3 项目，通过 Vite 构建
- 开发阶段：前端 dev server 通过代理将 API 请求转发到后端
- 生产阶段：Nginx 统一托管前端静态资源，反向代理 `/api/` 和 `/sub/` 到后端
- `Nginx` 统一反向代理到 `app`，不再单独部署前端服务

### 4.2 前端技术实现

- 页面模板：Vue 3 单文件组件（SFC）
- UI 组件库：Element Plus，使用默认风格，轻度主题定制
- 异步请求：Alova，封装统一请求实例，处理鉴权、错误提示、分页
- DOM 与事件：Vue 3 组合式 API
- 状态管理：Pinia
- 路由：Vue Router 4，手动配置路由表
- 第三方前端脚本：优先本地 npm 安装，不依赖生产环境 CDN

### 4.3 页面范围

v1 页面范围固定为两类，单项目通过路由区分：

- 用户侧页面：登录页、用户首页、我的订阅、订单列表、套餐页、兑换码页；我的订阅页展示按天/周/月聚合的用量记录
- 后台页面：后台登录、后台首页、套餐管理、节点组管理、节点管理、用户管理、订单管理、兑换码管理；用户管理列表直接展示套餐与剩余流量，并可查看指定用户用量记录，支持多选批量删除非当前登录用户

建议页面路径：

- `/login` — 用户登录
- `/register` — 用户注册
- `/` — 用户首页
- `/subscription` — 我的订阅
- `/orders` — 订单列表
- `/plans` — 套餐列表
- `/redeem` — 兑换码
- `/admin/login` — 后台登录
- `/admin` — 后台首页
- `/admin/plans` — 套餐管理
- `/admin/node-groups` — 节点组管理
- `/admin/nodes` — 节点管理
- `/admin/users` — 用户管理，列表直接展示套餐、订阅状态和剩余流量，支持多选批量删除
- `/admin/orders` — 订单管理
- `/admin/redeem-codes` — 兑换码管理

### 4.4 浏览器鉴权与交互规则

- 登录成功后，服务端返回 Access Token（JWT）和设置 Refresh Token（HttpOnly Cookie）
- 前端将 Access Token 存储在内存中，通过 Alova 请求拦截器自动注入到 Header
- Refresh Token 由浏览器自动携带，前端不可读取（防 XSS）
- 非浏览器调用（如 node-agent）仍可保留独立 token 鉴权
- 敏感写操作接口需要 CSRF 防护（通过 SameSite Cookie + 自定义 Header 双重防护）

页面交互规则：

- 首屏由 Vue SPA 加载
- 列表、表格、表单、弹窗优先通过 Element Plus 组件实现
- 页面与后端的交互统一通过 Alova 请求 REST API
- 不在浏览器本地存储完整业务状态

### 4.5 页面与接口的关系

- 页面路由由 Vue Router 管理，API 路由由 Gin Router 注册
- 所有业务数据通过 JSON API 交互
- 前端应为 Element Plus 表格、表单等组件封装统一适配层，不让后端接口结构被单一组件默认格式绑死
- 需要复用给未来外部客户端的能力，仍然保留 REST API

## 5. 数据库设计

### 5.1 设计调整

相比旧版方案，关键调整如下：

- 删除 Plan.nodeGroups Int[]，增加 PlanNodeGroup 关联表
- 增加 SubscriptionToken，专门用于订阅下载
- 增加 Order，支付记录和业务订单拆开
- 流量统计拆成 TrafficSnapshot 和 UsageLedger
- Node 不再直接暴露给服务端调用 xray-core gRPC，统一改为记录 node-agent 连接信息
- 增加 NodeAccessTask，把订阅状态变化和节点侧用户同步解耦
- UsageLedger 增加对订阅和节点的外键关系，保证账本可追溯
- User 增加 xray_user_key，统一作为 xray-core 统计标识
- 新增 RedeemCode 表，支持兑换码开通订阅
- 为适配 MySQL 的唯一约束能力，UserSubscription 增加 active_user_id 作为活动订阅唯一辅助列
- users 增加 last_login_at、last_login_ip 字段
- nodes 和 plans 增加 sort_weight 字段，控制展示顺序
- plans 增加 is_default 与 is_deleted 字段，基础套餐不可删除，普通套餐删除采用逻辑删除并迁移活动订阅到基础套餐
- Node 与 NodeGroup 改为 `node_group_nodes` 多对多关联，节点归组统一在节点分组管理中维护；`nodes.node_group_id` 仅作为历史兼容字段读取
- `nodes` 在后续文档和界面中统一称为出口节点，保留现有 node-agent、用户同步和流量账本职责
- 新增 `relays` 与 `relay_backends`，用于描述中转入口服务器和中转端口到出口节点的绑定
- 新增 `relay_config_tasks`，用于 node-agent relay 模式拉取配置变更、渲染转发配置并 reload

### 5.2 Go 模型与 MySQL 表结构

落地方式：

- 数据表结构通过 SQL migration 维护
- Go 侧使用 GORM 模型映射表结构
- MySQL 版本固定为 8.0+，字符集统一为 utf8mb4，存储引擎统一为 InnoDB
- 复杂查询、租约锁、批量更新允许直接使用原生 SQL

核心表如下：

**users（用户表）**

关键字段：id、uuid、username、password_hash、email、xray_user_key、status、is_admin、last_login_at、last_login_ip、created_at、updated_at

**plans（套餐表）**

关键字段：id、name、price、currency、traffic_limit、duration_days、sort_weight、is_active、is_default、is_deleted、created_at、updated_at

说明：`is_default=true` 的记录是基础套餐。基础套餐不可删除，只能修改，并强制保持启用。普通套餐删除时设置 `is_deleted=true` 并从前后台可选列表隐藏，活动订阅自动迁移到基础套餐。

**node_groups（节点分组表）**

关键字段：id、name、description、created_at、updated_at

**plan_node_groups（套餐-节点分组关联表）**

关键字段：id、plan_id、node_group_id、created_at
约束：(plan_id, node_group_id) 唯一

**node_group_nodes（节点分组-节点关联表）**

关键字段：id、node_id、node_group_id、created_at
约束：(node_id, node_group_id) 唯一

**nodes（节点表）**

关键字段：id、name、protocol、transport、host、port、server_name、public_key、short_id、fingerprint、flow、line_mode、xhttp_path、xhttp_host、xhttp_mode、node_host_id、listen_ip、outbound_ip、xray_inbound_tag、xray_outbound_tag、agent_base_url、agent_token_hash、agent_version、last_heartbeat_at、last_traffic_report_at、last_traffic_success_at、is_enabled、sort_weight、created_at、updated_at

说明：`nodes` 是出口节点表。出口节点是真正运行 `xray-core` 的 Reality 终点，继续由 `node-agent` 管理用户开通、禁用和流量上报。`transport` 当前支持 `tcp` 和 `xhttp`，默认 `tcp`。`transport=xhttp` 时使用 VLESS + Reality + XHTTP，`flow` 必须为空，XHTTP 参数由 `xhttp_path`、`xhttp_host`、`xhttp_mode` 控制。`line_mode` 字段控制订阅输出策略，取值为 `direct_only`、`relay_only`、`direct_and_relay`，分别表示仅直连、仅中转、直连与中转同时下发。

多 IP 模式下，`nodes` 表示逻辑出口节点。一个公网出口 IP 对应一条 `nodes` 记录，`listen_ip` 是 Xray inbound 监听 IP，`outbound_ip` 是 freedom outbound 的 `sendThrough`，二者默认等于该节点 `host`。

**node_hosts（物理节点服务器表）**

关键字段：id、name、ssh_host、ssh_port、agent_base_url、agent_token_hash、agent_version、last_heartbeat_at、is_enabled、created_at、updated_at

说明：`node_hosts` 表示一台物理服务器和唯一 node-agent 身份。单 IP 部署可继续只使用 `nodes`；多 IP 部署创建一条 `node_hosts`，再为管理员确认的每个公网出口 IP 创建一条 `nodes`，这些节点通过 `nodes.node_host_id` 归属到同一台物理服务器。

**relays（中转节点表）**

关键字段：id、name、host、forwarder_type、agent_base_url、agent_token_hash、agent_version、status、last_heartbeat_at、is_enabled、created_at、updated_at

说明：中转节点只代表入口服务器和转发控制面，不承载用户 UUID，不参与订阅鉴权，不直接写入用户级流量账本。`agent_token_hash` 保存 `RELAY_TOKEN` 的哈希，只允许 node-agent relay 模式访问中转任务接口。`forwarder_type` 第一版固定默认 `haproxy`，后续可扩展 `nginx_stream` 或 `gost`。

**relay_backends（中转后端绑定表）**

关键字段：id、relay_id、exit_node_id、name、listen_port、target_host、target_port、is_enabled、sort_weight、created_at、updated_at

说明：一条记录就是订阅里的一条中转线路。`listen_port` 是客户端连接中转节点的端口，`target_host/target_port` 指向出口节点的 Xray 端口。`target_host` 默认可取出口节点 `nodes.host`，保留字段是为了支持内网地址或专线地址。

**relay_config_tasks（中转配置任务表）**

关键字段：id、relay_id、action、payload、status、retry_count、last_error、scheduled_at、locked_at、lock_token、executed_at、idempotency_key

说明：用于 node-agent relay 模式主动拉取中转配置任务，第一版渲染 HAProxy 配置并 reload。虽然复用同一个 node-agent 二进制，出口模式和中转模式仍必须使用不同 Token、不同任务表和不同权限范围。Nginx stream 和 gost 只作为后续转发器适配，不进入第一版默认实现。

**user_subscriptions（用户订阅表）**

关键字段：id、user_id、plan_id、start_date、expire_date、traffic_limit、used_traffic、status、active_user_id、created_at、updated_at

**subscription_tokens（订阅 Token 表）**

关键字段：id、user_id、subscription_id、token、is_revoked、last_used_at、expires_at、created_at

说明：订阅 Token 是用户级凭证，`subscription_id` 允许为空，仅作为最近关联订阅的辅助信息。每个用户只保留一条 Token 记录，`user_id` 有唯一约束。用户注册或管理员新增用户时即预生成 Token，并同时关联基础套餐订阅；续费、换套餐和重新开通订阅不会改变 Token，只有管理员重置 Token 时才会在同一条记录上更新 token 值。

**orders（订单表，v1 保留骨架）**

关键字段：id、order_no、user_id、plan_id、amount、currency、pay_address、expected_chain、status、paid_at、expired_at、created_at、updated_at

**payment_records（支付记录表，v1 保留骨架）**

关键字段：id、order_id、user_id、tx_id、chain、amount、from_address、to_address、raw_payload、status、confirmed_at、created_at

**traffic_snapshots（流量快照表）**

关键字段：id、node_id、xray_user_key、uplink_total、downlink_total、captured_at

**node_access_tasks（节点访问同步任务表）**

关键字段：id、node_id、subscription_id、action、payload、status、retry_count、last_error、scheduled_at、locked_at、lock_token、executed_at、idempotency_key

**usage_ledgers（流量账本表）**

关键字段：id、user_id、subscription_id、node_id、delta_upload、delta_download、delta_total、recorded_at

**redeem_codes（兑换码表）**

关键字段：id、code、plan_id、duration_days、is_used、used_by_user_id、used_at、expires_at、created_at

### 5.3 关键约束

- subscription_tokens.token 唯一
- subscription_tokens.user_id 唯一，一个用户只允许一条订阅 Token 记录
- orders.order_no 唯一
- payment_records.tx_id 唯一
- users.xray_user_key 唯一
- redeem_codes.code 唯一
- 同一用户同一时刻只允许一个 ACTIVE 订阅，MySQL 中通过 active_user_id 唯一索引或等效辅助列实现
- node_access_tasks 需要按 status + scheduled_at、status + locked_at 建索引
- traffic_snapshots 需要按 node_id + xray_user_key + captured_at 建索引
- usage_ledgers 需要按 subscription_id + recorded_at 建索引
- node_access_tasks.idempotency_key 需要唯一索引，避免重复下发
- relays.host 建议建普通索引，便于按入口地址查询
- relay_backends 需要 `(relay_id, listen_port)` 唯一索引，防止同一中转服务器端口冲突
- relay_backends.exit_node_id 需要索引，便于按出口节点查找可用中转线路
- relay_config_tasks.idempotency_key 需要唯一索引，避免重复下发中转配置
- 任务相关 SQL 允许直接使用 FOR UPDATE SKIP LOCKED，要求 MySQL 8.0+

## 7. 出口节点、中转与订阅生成

### 7.1 节点运行方式

每台出口节点统一部署：

- `xray-core`：负责实际代理流量转发
- `node-agent`：负责管理本机用户、调用本地 xray-core 能力、上报心跳和流量数据

每台中转节点部署：

- HAProxy：第一版默认 TCP 转发组件，负责端口透传
- `node-agent` relay 模式：负责从中心服务拉取转发配置任务、渲染配置、reload 转发组件、上报心跳和执行结果

多出口 IP 服务器部署：

- `node_hosts` 记录物理服务器和唯一 node-agent 身份
- `nodes` 记录逻辑出口节点，一个公网出口 IP 一条记录
- `node-agent` 以 `AGENT_ROLE=multi_exit` 运行，使用 `NODE_HOST_ID` + `NODE_HOST_TOKEN` 鉴权
- `MULTI_NODE_CONFIG` 描述该 agent 管理的多个 `node_id`、监听 IP、出站 IP、inbound tag 和 outbound tag
- Xray 使用一份配置，包含多个 VLESS Reality inbound 和多个 freedom outbound；每个 inbound 绑定对应 IP，每个 outbound 使用 `sendThrough` 绑定同一 IP
- 一键部署前必须清理目标服务器旧 systemd `node-agent`、旧 node-agent 二进制、旧 `raypilot-node-agent`/`suiyue-node-agent`/`raypilot-relay-agent`/`suiyue-relay-agent` 容器，并停止宿主机 `xray` 服务和进程释放 443；保留 `/usr/local/etc/xray` 配置目录用于 Reality 参数迁移

控制面约束：

- 中心服务对出口节点只调用 node-agent，不直接调用 xray-core
- node-agent 对本机 xray-core 的实现细节负责
- 订阅生成只负责给客户端下发"应该连哪些节点"，真正能否使用由 node-agent 是否已把用户写入本机决定
- 一键部署完成后，中心服务必须读取节点 Xray Reality 配置并写回 `server_name`、`public_key`、`short_id`；订阅下发字段必须与节点实际 `realitySettings` 一致，否则客户端可导入但无法握手
- multi_exit 模式下中心仍按 `nodes.id` 分发用户同步任务和写入流量账本；node-agent 必须按 `node_id` 分别心跳、领取任务、上报任务结果和流量
- multi_exit 写入 Xray clients 时使用节点内部分隔的统计 email，避免多个逻辑节点共享同一 `xray_user_key` 导致 Xray Stats 累计值混合；上报中心前再还原成原始 `xray_user_key`
- node-agent relay 模式只管理中转端口转发，不管理用户 UUID，不采集用户级流量，不参与订阅 Token 鉴权
- 中转节点第一版只支持 HAProxy 配置模板；Nginx stream 和 gost 仅保留为后续可选适配

### 7.2 当前支持的节点格式

当前默认使用：

- VLESS
- Reality
- TCP
- flow: xtls-rprx-vision

可选支持：

- VLESS
- Reality
- XHTTP
- flow: 空
- XHTTP 参数：`xhttp_path` 默认 `/raypilot`，`xhttp_mode` 默认 `auto`，可选 `packet-up`、`stream-up`、`stream-one`，`xhttp_host` 可选

XHTTP 仍然使用 Reality 作为安全层，不需要为 XHTTP 单独管理用户 UUID。客户端必须使用支持 VLESS + Reality + XHTTP 的内核或客户端；旧版 Clash 内核不支持 VLESS，mihomo 或 Xray 客户端需要支持 XHTTP 才能导入使用。

### 7.3 直连与中转线路生成规则

订阅生成必须先以出口节点为授权核心，再派生直连线路和中转线路。

**直连线路**

- `server` = `nodes.host`
- `port` = `nodes.port`
- `uuid` = 用户 `xray_user_key` 或对应 UUID
- `sni/pbk/sid/fingerprint/flow/transport/xhttp-*` = 出口节点 Reality 与传输层参数；XHTTP 线路不下发 `flow`
- 名称建议：`{出口节点名称} 直连`

**中转线路**

- `server` = `relays.host`
- `port` = `relay_backends.listen_port`
- `uuid` = 同一用户 UUID
- `sni/pbk/sid/fingerprint/flow/transport/xhttp-*` = 对应出口节点 Reality 与传输层参数；XHTTP 线路不下发 `flow`
- `target` = 中转节点透传到 `relay_backends.target_host:relay_backends.target_port`
- 名称建议：`{中转节点名称} -> {出口节点名称}`

中转线路不能使用中转节点自己的 Reality 参数，因为中转节点不终止 Reality 握手。客户端看到的是中转入口地址，但握手校验的是出口节点的 Reality 身份。

中转线路不需要额外用户 Token。客户端连接时携带的用户 UUID 会被 TCP 透传到出口节点，出口节点 xray 仍按该 UUID 做用户校验，node-agent exit 模式仍按该 UUID 做用户级流量统计和套餐扣量。node-agent relay 模式只能做按中转节点、监听端口、后端绑定的线路级统计。

### 7.4 订阅生成流程

客户端请求订阅接口（clash/base64/plain）后：

- 校验订阅 token
- 查询当前有效订阅
- 查询套餐允许访问的节点组
- 查询节点组下启用中的出口节点
- 按出口节点 `line_mode` 生成直连线路
- 查询指向这些出口节点且启用的 `relay_backends`
- 按 `relay_backends` 生成中转线路
- 按所需格式生成订阅内容
- 直接返回

建议：

- 订阅内容做 30 到 60 秒缓存
- token 命中记录 lastUsedAt
- 可加 ETag / Last-Modified
- 订阅下发和节点开通是两条链路：前者面向客户端，后者通过 NodeAccessTask 落地到节点
- 订阅发布验收不能只看“客户端可导入”，必须确认节点实际可连通；Reality 节点至少验证 `sni/pbk/sid` 与节点配置一致，并通过真实客户端或 Xray 本地 SOCKS 代理访问外部地址
- 中转线路发布验收必须额外确认中转入口端口可达、出口节点防火墙允许中转 IP、订阅中转线路的 `server/port` 与 `sni/pbk/sid` 分别来自正确的中转和出口记录

### 7.5 节点访问同步流程

以下事件触发写入 NodeAccessTask：

- 订阅开通
- 订阅续费
- 订阅到期
- 订阅超额

同步流程：

- 事件发生，写入 node_access_tasks 表
- node-agent 定时主动上报心跳时，服务端在响应中返回待执行任务列表
- node-agent 领取任务后执行，调用本机 xray-core 完成用户开通或禁用
- node-agent 上报执行结果，服务端回写任务状态
- 失败则记录错误并按重试策略重新排队

关键规则：

- 远程调用 node-agent 不放在数据库事务里
- 到期或超额后，不能只让订阅下载失效，还必须下发 DISABLE_USER
- 节点长期离线时，任务继续重试，直到节点恢复或人工介入
- 任务认领和执行基于 node-agent 主动拉取，不依赖服务端主动推送
- 超过重试次数仍未完成的任务，标记为失败，等待人工介入

### 7.6 中转配置同步流程

以下事件触发写入 RelayConfigTask：

- 创建、更新、删除中转节点
- 新增、更新、删除中转后端绑定
- 启用或禁用中转节点
- 启用或禁用某条中转线路

同步流程：

- 管理后台保存中转配置
- 服务端在事务内写入 `relays`、`relay_backends` 和 `relay_config_tasks`
- node-agent relay 模式心跳时拉取待执行任务
- node-agent relay 模式渲染 HAProxy 配置
- node-agent relay 模式执行 `haproxy -c` 测试配置合法性，reload HAProxy
- node-agent relay 模式上报任务结果

关键规则：

- node-agent relay 模式配置失败时，不影响出口节点 `node-agent` exit 模式的用户同步
- 中转线路不可用时，直连线路是否保留由出口节点 `line_mode` 决定
- 如果出口节点设置为 `relay_only`，必须确保至少存在一条健康中转线路，否则订阅会没有可用线路
- 中转配置任务也必须具备幂等键、重试次数、失败原因和人工重试能力
- HAProxy reload 失败时必须保持上一份可用配置，不得写入半成品配置导致已有中转线路中断

## 8. 流量统计设计

### 8.1 接入方式

推荐方式：

- node-agent 全量主动推送：agent 定时调用本机 Xray Stats API，并向中心 `/api/agent/traffic` 上报当前每个 xray_user_key 的累计上行/下行流量

说明：

- node-agent 可以在节点本地调用 Xray Stats API，默认地址为 `127.0.0.1:10085`
- 中心服务不直接连各节点的 xray-core
- 服务端不需要额外调度拉取任务，agent 自行定时上报
- node-agent 启动时需要自动补齐旧 Xray 配置的 Stats API 能力；一键部署出口节点必须传入 `XRAY_API_SERVER=127.0.0.1:10085`
- node-agent 每轮采集后先写入本地流量队列文件，再向中心上报；如果中心不可用，队列会保留采集结果，恢复后按 `collected_at` 顺序补发。
- 中心 `nodes` 表记录 `last_traffic_report_at`、`last_traffic_success_at`、`last_traffic_error_at`、`traffic_error_count` 和 `last_traffic_error`，用于后台展示和运维告警。`last_traffic_report_at` 表示中心最近一次收到报告的时间，`last_traffic_success_at` 表示节点最近一次成功处理到的采集时间。

### 8.2 快照差值法

node-agent 每轮上报时执行：

1. 读取节点当前每个 xray_user_key 的累计上行/下行
2. 将本轮采集结果和 `collected_at` 写入 node-agent 本地队列
3. 按队列顺序向中心 `/api/agent/traffic` 上报
4. 中心读取上一次该节点该用户的快照
5. 用本次累计值减上次累计值，得到增量
6. 按 `collected_at` 写入 TrafficSnapshot
7. 按 `collected_at` 写入 UsageLedger
8. 原子更新 UserSubscription.usedTraffic

### 8.3 关键规则

- 增量小于 0 时按节点重启处理，忽略负值并重建基线
- 同一节点在同一时刻只允许一个流量同步任务执行，避免重复计费
- 用户订阅过期或超额后，需要写入 NodeAccessTask(DISABLE_USER)，而不只是从订阅中剔除
- 不能依赖客户端主动同步
- 流量同步由 node-agent 主动上报触发，服务端在收到上报数据时同步处理
- node-agent 补发历史采集结果时，中心必须按请求中的 `collected_at` 作为快照和账本时间，不能使用补发到达时间覆盖采集时间
- 如果补发批次的 `collected_at` 早于或等于中心已有最新快照时间，中心必须幂等跳过该旧批次，避免乱序补发造成重复计费或错误重建基线
- 中心服务短时不可用时，只要节点 Xray 累计计数未重置，恢复后可通过本地队列和快照差值补账；如果 Xray 在失败期间重启且计数器归零，已清空且未被采集到的流量无法凭空恢复
- xray-core 统计标识统一使用 users.xray_user_key，由 node-agent 写入本机 xray-core 用户配置
- 只有中心服务更新不够；已有出口节点必须同步更新 node-agent 镜像或二进制，否则节点心跳会正常但 `traffic_snapshots` 和 `usage_ledgers` 不会增长

### 8.4 中转流量统计边界

中转模式下，用户级计费仍以出口节点为准：

- 客户端连接中转时携带用户 VLESS UUID
- 中转节点只做 TCP 透传，不解密 Reality，也不解析 UUID
- 出口节点 xray 最终识别用户 UUID
- node-agent exit 模式继续按 `xray_user_key` 上报用户级流量
- 套餐扣量、超额封禁、用户禁用仍以出口节点用户级统计为准

node-agent relay 模式可以做线路级统计，用于运维、质量分析和容量规划：

- `relay_id`
- `relay_backend_id`
- `listen_port`
- 端口级入站/出站字节
- 中转到出口的连通状态

node-agent relay 模式通过 HAProxy stats socket 读取 `show stat` 的累计计数，默认 socket 路径为 `/tmp/haproxy.sock`，可通过 `HAPROXY_STATS_SOCKET_PATH` 覆盖。中心服务将上报结果写入 `relay_traffic_snapshots`，但这些数据不能直接作为用户套餐扣量依据。若未来要求中转侧精确识别用户，需要采用每用户独立端口或中转终止协议等方案；第一版不采用这些高复杂度方案。

## 10. 安全设计

### 10.1 鉴权

- 后台接口：JWT（Access Token + Refresh Token）
- 订阅接口：高熵随机 token
- 节点 agent：每节点独立凭证，服务端只保存凭证哈希
- 管理接口：isAdmin + 审计日志
- Access Token 存于前端内存，Refresh Token 存于 HttpOnly Cookie

### 10.2 限流

- /api/auth/login 做 IP 限流
- /sub/:token/* 做基础频控
- /api/payment/callback 做来源校验和限流

### 10.3 数据安全

- 密码用 bcrypt 哈希存储
- 不在日志打印完整 token
- 不在日志打印敏感支付回调字段原文
- 不在数据库明文保存 agent 凭证
- 配置文件中的节点私钥禁止进库，只存公钥侧信息

## 11. 部署方案

### 11.1 整体部署架构

Docker Compose 包含以下服务：

- **mysql**：MySQL 8.4 数据库
- **api**：后端 Go 应用，提供 API 和订阅接口
- **worker**：后台 Worker，处理定时任务
- **frontend**：Vue 3 构建产物，Nginx 托管静态资源

Nginx 统一反向代理：
- 根路径 `/` 和前端资源指向 frontend 静态文件
- `/api/` 反向代理到后端 API 服务
- `/sub/` 反向代理到后端订阅接口

### 11.2 docker-compose 说明

- 数据库 migration 不建议放在镜像构建阶段，建议作为发布步骤单独执行
- 每台节点需额外部署 xray-core + node-agent
- 中心服务只暴露业务 API，不直接暴露到节点 xray-core
- 中心服务部署只依赖 app + mysql，v1 不额外引入 Redis

## 15. 冻结结果与剩余风险

本版方案已冻结的业务决策：

1. 订阅 token 允许多端共用
2. 套餐续费采用时长叠加，不重置已用流量
3. 超流量后立即封禁节点访问
4. xray_user_key 为每用户全局稳定标识，不按订阅轮换
5. node-agent 与中心服务采用全量主动推送模式
6. node-agent 鉴权采用独立 token
7. v1 采用前后端分离架构，Vue 3 + Element Plus
8. v1 支付功能保留骨架暂不上线
9. 支持三种订阅格式：Clash YAML、Base64 聚合、纯文本 URI
10. 新增兑换码模块支持开通订阅
11. 保留现有直连模式，新增中转模式时不破坏 node-agent 用户同步和流量计费
12. 一个用户仍只有一个订阅 Token，直连和中转只是同一订阅内容里的多条线路
13. 中转节点只做 TCP 透传，不做 Reality 终止、用户鉴权或用户级流量计费
14. 中转节点使用 `RELAY_TOKEN` 访问中心服务；客户端用户不需要中转专用 Token，用户 UUID 会透传到出口节点
15. 中转节点第一版默认使用 HAProxy 做 TCP 透传，Nginx stream/gost 只作为后续可选适配

剩余工程风险：

- node-agent 协议字段级定义仍需继续细化到请求体和错误码层级
- xray_user_key 在 xray-core 本地具体写入字段需在 node-agent 实现时再做联调验证
- traffic_snapshots、usage_ledgers 的保留策略仍需在真实数据量下调整
- 如果支付提供方无法支持订单唯一地址，则仍需补充替代对账方案
- Refresh Token 的续期策略和过期处理需要在实际联调中验证
- 中转能力已完成第一版实现，仍需持续用真实中转服务器压测 HAProxy reload、stats socket 上报和 Worker 重试表现
- 隐藏出口 IP 依赖出口节点防火墙策略，不能只靠订阅不展示直连线路
- 同一中转端口多出口负载均衡会受 Reality 身份一致性限制，默认不支持
- API/UI 主要页面、管理后台主路径、多用户路径和流量账本主路径已通过 Playwright 验证，但尚未穷举所有破坏性 CRUD 操作组合
- 支付模块仍是骨架能力，正式收款前必须补齐回调验签、幂等、对账和异常补偿测试
- 当前验收结论是核心闭环可用，不等于零缺陷；生产前仍需长时间稳定性观察和并发压测
