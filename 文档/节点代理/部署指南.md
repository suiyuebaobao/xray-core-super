# 节点代理（node-agent）部署指南

## 概述

`node-agent` 是部署在每台代理节点上的控制面程序，负责：

- **自动安装 xray-core**（启动时检测，未安装则自动安装）
- 管理本机 `xray-core` 的用户配置（增删改）
- 定时向中心服务发送心跳，拉取待执行任务
- 采集流量数据并上报中心服务
- 执行用户开通/禁用等任务

本文档同时适用于出口节点和中转节点。中转节点复用同一个 `node-agent` 二进制，但以 `AGENT_ROLE=relay` 运行，只管理 TCP 转发端口和中转配置任务，不管理用户 UUID。中转线路的用户 UUID、Reality 参数和用户级流量统计仍然落在出口节点。

运行角色：

- `AGENT_ROLE=exit`：出口节点模式，使用 `NODE_ID` + `NODE_TOKEN`
- `AGENT_ROLE=multi_exit`：单服务器多出口 IP 模式，使用 `NODE_HOST_ID` + `NODE_HOST_TOKEN` + `MULTI_NODE_CONFIG`，一个 agent 管理多个逻辑出口节点
- `AGENT_ROLE=relay`：中转节点模式，使用 `RELAY_ID` + `RELAY_TOKEN`，第一版默认管理 HAProxy TCP 转发配置

## 出口模式自动安装流程

node-agent 启动后自动执行以下步骤：

```
1. 检查 xray 是否已安装（执行 xray version）
   │
   ├── 已安装 → 检查配置文件是否存在 → 不存在则生成
   │           补齐 xray Stats API 配置 → 检查 systemd 服务是否存在 → 不存在则安装
   │
   └── 未安装 → 执行自动安装
                │
                ├── 方案 A：官方安装脚本（install-release.sh）
                │    下载脚本 → 执行安装 → 验证
                │
                └── 方案 B：手动安装（方案 A 失败时）
                     检测 CPU 架构 → 查询最新 GitHub Release →
                     下载对应 zip → 解压到 /usr/local/bin/xray
                │
                └── 生成默认配置
                     执行 xray x25519 生成 Reality 密钥对 →
                     写入 /usr/local/etc/xray/config.json
                │
                └── 安装 systemd 服务
                     写入 /etc/systemd/system/xray.service →
                     daemon-reload → enable → start
```

安装完成后，node-agent 继续正常启动流程。出口模式会确保 Xray 配置包含 `api`、`stats`、`policy.levels.0.statsUserUplink/statsUserDownlink` 和本地 `dokodemo-door` API 入站，默认监听 `127.0.0.1:10085`，用于 `xray api statsquery` 读取用户级累计流量。

## 配置方式

通过环境变量配置，无需额外配置文件。

### 必要参数

| 环境变量 | 说明 | 示例 |
|----------|------|------|
| `CENTER_SERVER_URL` | 中心服务 API 地址 | `https://api.example.com` |
| `NODE_ID` | 节点 ID（在管理后台创建后获取） | `1` |
| `NODE_TOKEN` | 节点鉴权 Token（在管理后台创建时设置） | `your-node-token` |

### 可选参数

| 环境变量 | 说明 | 默认值 |
|----------|------|--------|
| `XRAY_CONFIG_PATH` | xray-core 配置文件路径 | `/etc/xray/config.json` |
| `XRAY_RESTART_CMD` | 重启 xray-core 的命令 | `systemctl restart xray` |
| `XRAY_BINARY` | xray 可执行文件路径 | `xray` |
| `XRAY_API_SERVER` | xray API 地址（用于流量统计） | `127.0.0.1:10085` |
| `NODE_TRANSPORT` | 单出口节点传输模式，`tcp` 或 `xhttp` | `tcp` |
| `XHTTP_PATH` | XHTTP 路径，仅 `NODE_TRANSPORT=xhttp` 有效 | `/raypilot` |
| `XHTTP_HOST` | XHTTP Host，仅 `NODE_TRANSPORT=xhttp` 有效 | 空 |
| `XHTTP_MODE` | XHTTP mode，可选 `auto`、`packet-up`、`stream-up`、`stream-one` | `auto` |
| `HEARTBEAT_INTERVAL` | 心跳上报间隔 | `10s` |
| `REPORT_INTERVAL` | 流量上报间隔 | `60s` |
| `TRAFFIC_QUEUE_PATH` | 流量上报失败队列文件路径 | 与 Xray 配置同目录的 `traffic_queue.json` |
| `TRAFFIC_QUEUE_LIMIT` | 本地流量队列最大批次数 | `720` |

出口模式每次采集 Xray 累计流量后，会先写入本地队列文件，再上报中心。中心不可用时队列不会丢弃，恢复后按采集时间顺序补发。若队列超过 `TRAFFIC_QUEUE_LIMIT`，最旧批次会被丢弃并在日志中记录。中心会按 `collected_at` 写入账本，且会跳过早于或等于已有最新快照的旧批次，避免断线恢复后的乱序补发重复计费。

### 中转模式参数

| 环境变量 | 说明 | 示例/默认值 |
|----------|------|-------------|
| `AGENT_ROLE` | 运行角色 | `relay` |
| `CENTER_SERVER_URL` | 中心服务 API 地址 | `https://api.example.com` |
| `RELAY_ID` | 中转节点 ID | `1` |
| `RELAY_TOKEN` | 中转节点鉴权 Token | `your-relay-token` |
| `HAPROXY_CONFIG_PATH` | HAProxy 配置路径 | `/etc/haproxy/haproxy.cfg` |
| `HAPROXY_PID_PATH` | HAProxy PID 路径 | `/tmp/haproxy.pid` |
| `HAPROXY_STATS_SOCKET_PATH` | HAProxy stats socket 路径，用于线路级流量上报 | `/tmp/haproxy.sock` |
| `HEARTBEAT_INTERVAL` | 心跳上报间隔 | `10s` |
| `REPORT_INTERVAL` | 线路级流量上报间隔 | `60s` |

### 多出口 IP 模式参数

| 环境变量 | 说明 | 示例/默认值 |
|----------|------|-------------|
| `AGENT_ROLE` | 运行角色 | `multi_exit` |
| `CENTER_SERVER_URL` | 中心服务 API 地址 | `https://api.example.com` |
| `NODE_HOST_ID` | 物理节点服务器 ID | `1` |
| `NODE_HOST_TOKEN` | 物理节点服务器鉴权 Token | `your-node-host-token` |
| `MULTI_NODE_CONFIG` | 逻辑出口节点 JSON 数组 | `[{"node_id":1,"ip":"154.219.106.105","port":443}]` |
| `MULTI_NODE_CONFIG_PATH` | 逻辑节点配置文件路径；与 `MULTI_NODE_CONFIG` 二选一 | `/etc/raypilot/nodes.json` |
| `XRAY_CONFIG_PATH` | xray-core 配置文件路径 | `/usr/local/etc/xray/config.json` |
| `XRAY_API_SERVER` | xray API 地址 | `127.0.0.1:10085` |

multi_exit 模式会生成一份 Xray 配置，其中每个逻辑节点都有独立 VLESS Reality inbound 和 freedom outbound。inbound `listen` 绑定该节点 IP，outbound `sendThrough` 绑定同一 IP。agent 对中心按 `node_id` 分别领取用户同步任务、上报任务结果和用户级流量。

`MULTI_NODE_CONFIG` 支持为每个逻辑节点指定 `transport`、`xhttp_path`、`xhttp_host`、`xhttp_mode`：

```json
[
  {
    "node_id": 33,
    "ip": "154.219.106.105",
    "port": 443,
    "transport": "xhttp",
    "xhttp_path": "/raypilot",
    "xhttp_mode": "auto",
    "inbound_tag": "node_33_in",
    "outbound_tag": "node_33_out"
  }
]
```

XHTTP 节点仍使用 VLESS + Reality 作为协议和安全层，但 Xray client 不写入 `xtls-rprx-vision` flow。TCP 节点继续使用 `flow=xtls-rprx-vision`。

## 部署步骤

### 1. 在管理后台创建节点

在管理后台「节点管理」页面添加节点，记录：
- **节点 ID**
- **Agent Token**（明文，用于节点鉴权）

### 2. 上传 node-agent 二进制

```bash
# 在开发机上构建
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o node-agent ./cmd/node-agent

# 上传到节点服务器
scp node-agent root@your-node:/usr/local/bin/node-agent
```

### 3. 创建 systemd 服务

```ini
# /etc/systemd/system/node-agent.service
[Unit]
Description=RayPilot Node Agent
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/node-agent
Restart=always
RestartSec=10
Environment="CENTER_SERVER_URL=http://your-center-api"
Environment="NODE_ID=1"
Environment="NODE_TOKEN=your-node-token"
Environment="XRAY_API_SERVER=127.0.0.1:10085"

[Install]
WantedBy=multi-user.target
```

### 4. 启动服务

```bash
systemctl daemon-reload
systemctl enable node-agent
systemctl start node-agent
systemctl status node-agent
```

启动后，node-agent 会自动检测并安装 xray-core（如果未安装），然后开始正常工作。

### 5. 验证

```bash
# 查看 node-agent 日志
journalctl -u node-agent -f

# 确认 xray 已自动安装
xray version

# 确认 xray 服务已自动启动
systemctl status xray
```

## 默认生成的 xray 配置

如果节点上没有 xray 配置文件，node-agent 会自动生成 VLESS + Reality + TCP 的默认配置：

- 监听端口：443
- 协议：VLESS + Reality
- 目标伪装：www.microsoft.com:443
- 解密：none
- 嗅探：开启（http, tls）
- Reality 密钥对：自动通过 `xray x25519` 生成
- 流量统计：默认启用 Xray Stats API，API 入站只监听 `127.0.0.1:10085`

## Reality 参数回写

一键部署完成后，中心服务会读取节点 `/usr/local/etc/xray/config.json`，把 Reality 连接参数写回 `nodes` 表：

- `server_name`：取 `realitySettings.serverNames[0]`
- `public_key`：优先取 `realitySettings.publicKey`；若配置中只有 `privateKey`，通过 `xray x25519 -i <privateKey>` 派生 PublicKey
- `short_id`：取 `realitySettings.shortIds[0]`

手动部署或手动修改 Xray 配置后，也必须同步更新中心节点记录。否则订阅可以导入，但客户端 Reality 握手会失败，表现为节点无信号。

验收时至少检查：

```bash
cat /usr/local/etc/xray/config.json
xray run -test -config /usr/local/etc/xray/config.json
```

订阅链接中的 `sni`、`pbk`、`sid` 必须分别等于节点 Xray 配置里的 `serverNames[0]`、PublicKey、`shortIds[0]`。

## 出口节点一键部署

管理后台的一键部署出口节点会推送 `raypilot/node-agent:latest` 镜像并以 host 网络启动容器。容器环境会显式设置：

```bash
-e XRAY_API_SERVER=127.0.0.1:10085
```

node-agent 启动时会补齐旧节点 `/usr/local/etc/xray/config.json` 中缺失的 Stats API 配置，并在容器内确保 Xray 进程已启动。更新已有出口节点时，必须重新构建 `/root/node-agent-image.tar.gz` 并重新部署或手动替换节点上的 node-agent 容器，否则中心服务只能看到心跳，无法收到 `/api/agent/traffic` 上报。

一键部署会先清理旧节点代理形态：停止并禁用旧 systemd `node-agent`，删除旧 service 文件和 `/usr/local/bin/node-agent` 等旧二进制，删除旧 `raypilot-node-agent`/`suiyue-node-agent`/`raypilot-relay-agent`/`suiyue-relay-agent` 容器，并停止宿主机 `xray` 服务和进程以释放 443 端口。Xray 配置目录 `/usr/local/etc/xray` 会保留，用于迁移 Reality 参数和客户端配置。

### 多出口 IP 一键部署

多出口 IP 服务器必须由管理员在一键部署前手动选择“多 IP 服务器”。未开启时，后端不得扫描出口 IP，也不得自动创建多个节点。

开启后流程如下：

1. 管理员点击“扫描出口 IP”。
2. 中心服务通过 SSH 执行 `ip -o -4 addr show scope global` 读取候选 IPv4。
3. 服务端过滤私网、回环、链路本地、组播等非公网地址。
4. 对每个公网候选地址执行 `curl --interface <IP>` 验证真实出口等于该 IP。
5. 前端展示扫描结果，管理员手动勾选要创建为节点的 IP。
6. 部署时创建一条 `node_hosts` 记录，并为每个选中 IP 创建一条 `nodes` 逻辑出口节点。
7. 部署前清理目标服务器旧 systemd node-agent、旧 node-agent 二进制、旧出口/中转 agent 容器和冲突的宿主机 Xray 进程。
8. 目标服务器只启动一个 `AGENT_ROLE=multi_exit` 的 `raypilot-node-agent` 容器。
9. agent 生成多 inbound/outbound Xray 配置，中心读取 Reality 参数并写回所有逻辑节点。
10. 管理员在节点分组页面把新逻辑节点加入对应分组，后端会为该分组下的活跃订阅触发用户同步任务。

multi_exit 验收必须确认：

- 每条 `nodes.host`、`nodes.listen_ip`、`nodes.outbound_ip` 对应同一个公网 IP。
- 目标服务器只有一个当前角色的 node-agent；不存在旧 systemd `node-agent`、旧 node-agent 二进制或旧 relay agent 容器。
- `/usr/local/etc/xray/config.json` 中每个逻辑节点 inbound 的 `listen` 绑定对应 IP。
- 每个逻辑节点 outbound 的 `sendThrough` 绑定对应 IP。
- `nodes.server_name/public_key/short_id` 与 Xray Reality 参数一致。
- 订阅下发包含所有已加入节点组的新逻辑节点，且不包含已停用的旧节点或旧中转线路。
- 使用对应订阅线路连接后，真实出站 IP 等于该逻辑节点 IP。

## 中转节点部署

推荐通过管理后台「中转节点」页面的一键部署执行。该流程会：

1. 通过 SSH 连接目标服务器。
2. 检测或安装 Docker。
3. 推送 `raypilot/node-agent:latest` 镜像。
4. 创建 `relays` 记录并生成 `RELAY_TOKEN`。
5. 以 `AGENT_ROLE=relay` 启动 `raypilot-relay-agent` 容器。
6. 容器内使用 HAProxy 管理 TCP 监听端口，配置目录挂载到 `/etc/raypilot/haproxy`。
7. 等待 relay agent 心跳回连；失败时清理本次中转记录和容器。

一键部署前，中心服务器需要准备最新版 node-agent 镜像包：

```bash
make node-agent-image
```

该命令会使用 `cmd/node-agent/Dockerfile` 从当前源码构建 `raypilot/node-agent:latest`，并导出 `/root/node-agent-image.tar.gz`，供一键部署上传到目标中转服务器。

手动运行示例：

```bash
docker run -d --name raypilot-relay-agent \
  --network host \
  --restart unless-stopped \
  -e AGENT_ROLE=relay \
  -e CENTER_SERVER_URL=https://api.example.com \
  -e RELAY_ID=1 \
  -e RELAY_TOKEN=your-relay-token \
  -e HAPROXY_CONFIG_PATH=/etc/haproxy/haproxy.cfg \
  -e HAPROXY_PID_PATH=/tmp/haproxy.pid \
  -e HAPROXY_STATS_SOCKET_PATH=/tmp/haproxy.sock \
  -v /etc/raypilot/haproxy:/etc/haproxy:rw \
  raypilot/node-agent:latest
```

保存中转后端绑定后，中心服务会创建 `RELOAD_CONFIG` 任务。node-agent relay 模式会先写临时 HAProxy 配置并执行 `haproxy -c -f <tmp>` 校验，成功后替换正式配置并 reload；失败时保留上一份可用配置。Worker 会定时把失败或锁超时的 `relay_config_tasks` 重新排队，直到达到重试上限。

relay 模式生成的 HAProxy 配置会启用 stats socket，node-agent 定时读取 `show stat` 并上报 `/api/agent/relay/traffic`。这只用于中转线路级容量和质量分析，不参与用户套餐扣量。

真实环境验收建议按以下顺序执行：

```bash
# 中心服务侧
curl -i http://your-center/health
curl -i http://your-center/api/plans

# 中转服务器侧
docker ps --filter name=raypilot-relay-agent
ss -lntp | grep 24443
docker exec raypilot-relay-agent haproxy -c -f /etc/haproxy/haproxy.cfg

# 中心数据库侧
SELECT status,last_heartbeat_at FROM relays WHERE id = <relay_id>;
SELECT status,last_error,executed_at FROM relay_config_tasks WHERE relay_id = <relay_id> ORDER BY id DESC LIMIT 1;
SELECT listen_port,bytes_in_total,bytes_out_total,captured_at FROM relay_traffic_snapshots WHERE relay_id = <relay_id> ORDER BY id DESC LIMIT 1;
```

订阅验收必须确认中转线路满足：

- `server` 是中转节点地址。
- `port` 是 `relay_backends.listen_port`。
- `uuid`、`sni`、`pbk`、`sid`、`fingerprint`、`flow` 沿用出口节点。
- 使用 Xray、mihomo 或真实客户端经中转线路能正常出站。

## 故障排查

### node-agent 无法连接中心服务

```bash
# 检查网络连通性
curl -v $CENTER_SERVER_URL/health

# 检查环境变量
systemctl status node-agent
journalctl -u node-agent -n 50
```

### xray 自动安装失败

查看日志中的 `[agent] downloading and installing xray-core` 部分：
- 如果官方脚本失败，会自动回退到手动安装
- 如果手动安装也失败，检查服务器是否能访问 GitHub

### xray 启动失败

```bash
# 检查配置是否合法
xray run -test -config /usr/local/etc/xray/config.json

# 查看 xray 日志
journalctl -u xray -f
```

### 流量统计始终为 0

按顺序检查：

```bash
# 节点侧确认 Xray Stats API 可用
xray api statsquery --server 127.0.0.1:10085 -pattern 'user>>>' -reset=false

# 节点侧确认 node-agent 使用新版镜像或二进制
docker ps --filter name=raypilot-node-agent
journalctl -u node-agent -n 100

# 中心侧确认 agent 上报已入库
SELECT COUNT(*) FROM traffic_snapshots;
SELECT COUNT(*) FROM usage_ledgers;
SELECT id,last_traffic_report_at,last_traffic_success_at,traffic_error_count,last_traffic_error FROM nodes;
```

如果节点心跳正常但两张流量表长期为 0，优先确认节点侧是否已经更新到包含 Stats API 自动补齐逻辑的 node-agent。旧 agent 可以代理和同步用户，但不会启用 `XRAY_API_SERVER`，因此不会产生用户级流量上报。

## 更新 node-agent

```bash
# 停止服务
systemctl stop node-agent

# 上传新版本
scp node-agent root@your-node:/usr/local/bin/node-agent

# 启动服务
systemctl start node-agent
systemctl status node-agent
```

```bash
journalctl -u node-agent -f
```
