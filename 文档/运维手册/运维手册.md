# 运维手册

本文档提供 suiyue（岁月）系统的日常运维、故障排查和应急响应指南。

## 系统架构概览

```
用户/客户端
    │
    ▼
Nginx (80/443) ← 静态资源 + API 反代
    │
    ├── /api/*  → 后端 API (:3000)
    ├── /sub/*  → 后端 API (:3000)
    └── /*      → 前端静态文件
                    │
                    ▼
               MySQL (容器内部，不暴露端口)

节点服务器
    │
    ├── xray-core (443) ← VLESS + Reality
    └── node-agent (systemd 服务)
              │
              ▼
         心跳 → 中心 API
         任务 ← 中心 API
         流量 → 中心 API

中转服务器
    │
    ├── HAProxy ← TCP 透传转发
    └── node-agent (AGENT_ROLE=relay)
              │
              ▼
         心跳 → 中心 API
         配置任务 ← 中心 API
         结果 → 中心 API
```

## 日常运维

### 服务启动/停止

```bash
# 启动所有服务
make up

# 停止所有服务
make down

# 仅启动 API 服务
make api

# 启动 Worker
make worker

# 启动前端开发服务器
make frontend
```

### 日志查看

```bash
# 查看所有服务日志
docker-compose logs -f

# 查看 API 日志
docker-compose logs -f api

# 查看 Worker 日志
docker-compose logs -f worker

# 查看 MySQL 日志
docker-compose logs -f mysql

# 查看 node-agent 日志（节点服务器）
journalctl -u node-agent -f

# 查看 xray-core 日志（节点服务器）
journalctl -u xray -f
```

日志中心 v1 口径：

- 后台 `运行日志` 页面读取宿主机 `logs/api/YYYY-MM-DD-HH-00.log` 与 `logs/worker/YYYY-MM-DD-HH-00.log`
- Docker Compose 部署会把宿主机 `./logs` 挂载到 API/Worker 容器 `/app/logs`，systemd 部署会设置 `RAYPILOT_LOG_DIR=/root/suiyue/logs`，两者都通过 `deploy/scripts/hourly-log.sh` 按小时切分 API/Worker 标准输出
- 新环境首次启动时日志文件尚未产生，运行日志接口返回空列表，不应返回 500；若页面无日志，先确认 `logs/` 目录权限、Compose 挂载或 systemd 环境变量，以及 `logs/{api,worker}/YYYY-MM-DD-HH-00.log` 是否生成
- 后台 `部署日志` 页面展示结构化的一键部署记录、步骤明细和失败原因
- 一键部署失败时，部署日志应包含镜像预检、服务器预检、Docker 安装/检查、容器启动、配置读取、心跳等待、端口验证等步骤；容器启动后的失败会附带 `docker ps`、最近容器日志和监听端口诊断
- 后台 `操作日志` 页面展示用户关键行为和管理员关键操作
- 结构化日志必须记录 `client_ip`，并保留 `X-Forwarded-For` / `X-Real-IP` 原始值用于反代链路排障
- 部署日志还必须记录 `operator_ip`、`target_server_ip`、`selected_exit_ips`、`relay_entry_ip`、`target_host`，出口节点部署只保存 `outbound_type`、`outbound_ip` 和 `outbound_proxy_count` 摘要，不保存完整 SOCKS5 账号密码
- 中心地址 SSH 兜底修复会记录为 `agent_center_repair` 部署日志，保存目标服务器、节点/物理主机/中转 ID、中心地址列表和步骤结果；不得保存 SSH 密码、完整 Token 或私钥
- 日志中不得写入密码、完整 Token、数据库连接串和 Reality 私钥

### Nginx 反代检查

Compose 内置 Nginx 通过 Docker DNS `127.0.0.11` 动态解析 `api` 服务名。滚动更新 API/Worker 后不需要停止项目，但要确认 Nginx 配置仍包含 `resolver 127.0.0.11` 与变量形式的 `proxy_pass`，避免 Nginx 缓存旧 API 容器 IP 后 `/api/*` 返回 502。

```bash
docker compose exec nginx nginx -t
docker compose exec nginx nginx -s reload
curl -i http://127.0.0.1/api/plans
```

### 服务健康检查

```bash
# 检查 API 健康
curl http://localhost:3000/health

# 检查 MySQL 连通性
mysql -u root -p -e "SELECT 1"

# 检查 node-agent 状态
systemctl status node-agent

# 检查 xray-core 状态
systemctl status xray

# 检查节点心跳（管理后台查看）
curl -H "Authorization: Bearer <TOKEN>" http://localhost:3000/api/admin/nodes
```

节点运营中心 v1：

- 后台路径 `/admin/node-operations`，接口为 `GET /api/admin/node-operations/summary`。
- worker 每分钟写入 `node_health_checks`，综合 node-agent 心跳、TCP 端口探测、最近流量上报、运行指标和 Xray 进程状态生成健康分。
- node-agent 新版心跳会带 `runtime_metric`，写入 `node_runtime_metrics`；旧 agent 未上报指标时，健康检查仍会按心跳、端口和流量状态判断。
- 健康状态只用于运营展示，不会自动禁用节点、不会自动隐藏订阅线路，也不会触发用户下线。
- TCP 可达只代表中心 worker 能连接 `nodes.host:nodes.port`，不等于真实 VLESS/Reality/XHTTP 客户端一定可用。客户端无信号仍需继续检查 Reality 参数、用户 UUID、Xray clients、订阅格式和真实客户端内核。
- 运营中心健康不代表远端 Xray 已包含所有可见活跃用户。若节点健康但客户端无信号，在后台 `/admin/nodes` 对节点服务器或单条线路点击“同步用户”，或调用 `POST /api/admin/nodes/:id/resync-users` 重新排队 `UPSERT_USER` 任务。

用户级限速 v1：

- 后台入口：`/admin/users`，点击用户“订阅”，在“用户限速(Mbps)”中设置；`0` 表示不限速。
- 数据库存储字段为 `user_subscriptions.speed_limit_bps`，单位 bps。
- 限速由出口节点 node-agent 执行，不改变订阅 URL，也不改变套餐流量扣费。中转节点不执行限速，经中转访问仍由出口节点控制。
- node-agent 默认每 5 秒读取 Xray Stats 计算用户平均速度；超过阈值时会临时从 Xray inbound 摘除该用户，默认 10 秒后恢复。该方式是滑动窗口平均限速，不是包级精准整形。
- 诊断限速策略：在节点服务器检查 `/usr/local/etc/xray/speed_limits.json`；不要把该文件提交到仓库。

健康原因排查：

| reason_code | 含义 | 优先处理 |
|-------------|------|----------|
| `server_unreachable` | 心跳过期且 TCP 不可达 | 检查服务器是否宕机、网络 ACL、安全组、本机防火墙 |
| `agent_offline` | TCP 可达但 agent 心跳过期 | 检查 `raypilot-node-agent` 容器或 systemd agent、中心地址、Token |
| `xray_or_firewall` | agent 在线但节点端口不可达 | 检查 Xray 监听、端口冲突、UFW/firewalld、安全组 |
| `xray_not_running` | agent 上报 Xray 未运行 | 检查 Xray 配置、容器日志、`xray run -test` |
| `traffic_report_stale` | 流量上报到达中心时间过旧 | 检查 Xray Stats API、本地流量队列、中心 `/api/agent/traffic`；空闲线路可能没有新增入账流量，优先看最近上报时间 |
| `traffic_report_error` | 最近流量上报处理失败 | 查看节点 `last_traffic_error` 和 API 日志 |
| `node_overloaded` | CPU/内存/磁盘/load 超阈值 | 检查并发连接、系统资源、异常进程 |

### 订阅与节点检查

- 订阅 URL 必须返回 200，且客户端导入的是完整链接 `/sub/{token}`，不是裸 Token；`/clash`、`/base64` 与 `/plain` 等后缀入口已下线。
- Reality 节点必须确认订阅里的 `sni/pbk/sid` 与出口节点 Xray 配置一致。
- 出口节点必须确认用户 UUID 已写入 Xray `clients`，并通过真实客户端或 Xray 本地代理完成一次出站访问。
- 多出口 IP 节点必须确认目标服务器只有一个当前角色的 node-agent，每个逻辑节点的 Xray inbound `listen` 与 outbound `sendThrough` 都绑定同一个公网 IP，订阅下发的每条线路真实出站 IP 等于对应逻辑节点 IP。
- 中转线路必须确认中转入口端口可达、中转到出口端口可达、`relay_config_tasks` 最新任务为成功状态、`relay_traffic_snapshots` 有流量记录。
- 已清理或停用的旧中转节点必须同步停用 `relays` 与 `relay_backends`；停用出口或改作管理系统的服务器也必须停用相关 `relay_backends` 并触发一次中转 `RELOAD_CONFIG`，确认 HAProxy 不再转发到旧目标。
- 如果要隐藏出口 IP，必须把出口节点设置为只下发中转线路，并在出口节点防火墙只放行中转服务器 IP 访问 Xray 端口。

### 尚未覆盖的风险

- 长时间稳定性观察和并发压测。
- 支付回调验签、幂等、对账和异常补偿全链路。
- 管理后台主路径新增、编辑、删除、禁用、恢复已由 Playwright 覆盖；复杂组合、长时间稳定性和并发压测仍需按生产发布单独验收。
- 多中转节点、多出口节点、大量用户同步任务下的 Worker 重试表现。

## 故障排查

### 控制平台迁移或中心地址不可达

当前控制平台应配置一个主入口和若干备用入口。真实域名和 IP 通过未提交的环境变量维护：

```bash
主入口: <primary-center-url>
备用入口: <fallback-center-url-a>
备用入口: <fallback-center-url-b>
```

一键部署出口节点和中转节点时，agent 容器必须同时具备：

```bash
CENTER_SERVER_URL=<primary-center-url>
CENTER_SERVER_URLS=<primary-center-url>,<fallback-center-url-a>,<fallback-center-url-b>
```

排查顺序：

```bash
# 1. 从节点服务器验证主备中心都能访问
curl -i <fallback-center-url-a>/health
curl -i <fallback-center-url-b>/health

# 2. 检查 Docker agent 的环境变量
docker inspect raypilot-node-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL
docker inspect raypilot-relay-agent --format '{{range .Config.Env}}{{println .}}{{end}}' | grep CENTER_SERVER_URL

# 3. 检查 systemd agent 的 drop-in
systemctl cat node-agent
systemctl cat raypilot-node-agent
```

如果节点无法自动切换中心，在后台执行：

- 出口节点：`/admin/nodes` -> 对应节点服务器 -> “修复中心”。
- 中转节点：`/admin/relays` -> 对应中转 -> “修复中心”。

修复入口会通过 SSH 登录节点服务器，重建 Docker agent 或写入 systemd drop-in，并可等待 `node_id`、`node_host_id` 或 `relay_id` 新心跳。等待超时表示中心地址已写入但还未确认新心跳，需要查看远端 agent 日志。不要在未知状态下手工删除运行中的 agent 容器；Docker 修复脚本会在新容器启动失败时恢复旧容器。

### 一键部署失败排查

一键部署出口节点和中转节点必须以 Docker 容器路径完成，不应在失败后手工 SSH 修改 Xray 或 HAProxy 配置来冒充成功。排查顺序：

```bash
# 中心侧确认镜像包存在
ls -lh deploy/artifacts/node-agent-image.tar.gz /root/node-agent-image.tar.gz
docker compose exec api ls -lh /root/raypilot-artifacts/node-agent-image.tar.gz

# 目标服务器侧确认 Docker 和容器状态
docker info
docker ps -a --filter name=raypilot-node-agent
docker ps -a --filter name=raypilot-relay-agent
docker logs --tail 160 raypilot-node-agent
docker logs --tail 160 raypilot-relay-agent
ss -lntp
```

出口部署成功标准：中心已写回 Reality 参数，node-agent 或 node_host 已产生新心跳，Xray 配置端口与后台 `nodes.port` 一致，监听进程是 `xray`，并且公网 TCP 端口可达。中转部署成功标准：relay agent 已产生新心跳，`RELOAD_CONFIG` 最新任务为成功，中转监听端口由 `haproxy` 监听，并且公网 TCP 端口可达。只看到端口被 nginx、宝塔或其他进程监听不能算成功。

一键部署会自动放行目标机本机防火墙的 TCP 监听端口：UFW 环境执行 `ufw allow <port>/tcp`，firewalld 环境执行 `firewall-cmd --permanent --add-port=<port>/tcp` 并 reload。若 Xray/HAProxy 已监听但客户端连接超时，优先检查云安全组或上游 ACL；若同机 `ss -lntp` 正常但公网 `nc -vz <ip> <port>` 超时，通常是防火墙或安全组问题。

### 用户反馈订阅失效

```sql
-- 检查用户订阅状态
SELECT us.id, us.user_id, us.status, us.expire_date, u.username
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE u.username = 'username';

-- 检查 Token 是否有效
SELECT token, user_id, subscription_id, is_revoked, expires_at, last_used_at
FROM subscription_tokens
WHERE token = 'token_string';

-- 检查 xray 配置中是否有该用户
-- 登录节点服务器执行：
cat /usr/local/etc/xray/config.json | grep "user_key"
```

如果订阅能导入但节点无信号，优先核对 Reality 参数：

```sql
SELECT id, name, host, port, server_name, public_key, short_id
FROM nodes
WHERE is_enabled = 1;
```

节点侧执行：

```bash
cat /usr/local/etc/xray/config.json
xray run -test -config /usr/local/etc/xray/config.json
systemctl status xray node-agent
```

`nodes.server_name` 必须等于 `realitySettings.serverNames[0]`，`nodes.public_key` 必须等于节点 PublicKey，`nodes.short_id` 必须等于 `shortIds[0]`。

如果 Reality 参数一致、端口可达但客户端仍无信号，继续确认用户 UUID 是否已经写入远端 Xray `clients`。后台节点列表的“同步用户”会按该节点当前所属分组重新补发可见活跃订阅的 `UPSERT_USER`，适合修复历史任务失败、节点配置被清理或后补节点分组后未同步的情况。

如果是中转线路，继续检查：

```bash
# 检查中转入口端口
nc -vz <RELAY_HOST> <LISTEN_PORT>

# 登录中转服务器，检查到出口节点的连通性
nc -vz <EXIT_NODE_HOST> 443

# 登录出口节点，确认只放行中转 IP 时防火墙规则生效
iptables -S
ufw status
```

中转线路的 `server/port` 来自中转节点，但 `sni/pbk/sid` 必须仍然来自出口节点。隐藏出口 IP 时，还必须在出口节点防火墙限制只允许中转服务器访问 Xray 端口。

### 流量统计异常

```sql
-- 检查用户当前订阅使用量（普通流量 + 家宽流量）
SELECT us.id, u.username,
       us.traffic_limit, us.used_traffic,
       ROUND(us.used_traffic * 100.0 / NULLIF(us.traffic_limit, 0), 2) AS normal_percent_used,
       us.residential_traffic_limit, us.residential_used_traffic,
       ROUND(us.residential_used_traffic * 100.0 / NULLIF(us.residential_traffic_limit, 0), 2) AS residential_percent_used
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE us.status = 'ACTIVE';

-- 检查最近流量快照
SELECT node_id, xray_user_key, uplink_total, downlink_total, captured_at
FROM traffic_snapshots
ORDER BY captured_at DESC
LIMIT 10;

-- 检查最近账本流量池归属
SELECT node_id, traffic_pool, xray_user_key, delta_total, recorded_at
FROM usage_ledgers
ORDER BY recorded_at DESC
LIMIT 20;
```

如果套餐启用了扣费倍率，排查时要同时看真实流量和扣费流量：

```sql
SELECT node_id, traffic_pool, billing_multiplier,
       delta_total AS actual_total,
       billed_total AS charged_total,
       recorded_at
FROM usage_ledgers
ORDER BY recorded_at DESC
LIMIT 20;
```

排查时必须额外确认：

- `nodes.traffic_pool` 是否符合该节点预期，普通节点应为 `normal`，家宽节点应为 `residential`
- 普通流量异常时看 `used_traffic`，家宽流量异常时看 `residential_used_traffic`
- `used_traffic` 与 `residential_used_traffic` 是扣费流量；真实节点流量看 `usage_ledgers.delta_total`，扣费量看 `usage_ledgers.billed_total`
- 单个流量池耗尽后，只应影响该池节点和对应中转线路，不应导致另一池节点一起消失

## 应急响应

### 数据库密码泄露

```bash
# 1. 修改 MySQL root 密码
docker-compose exec mysql mysql -u root -p -e "ALTER USER 'root'@'%' IDENTIFIED BY 'new_password'; FLUSH PRIVILEGES;"

# 2. 更新 DATABASE_URL 环境变量
# 编辑 .env 文件，更新 DATABASE_URL

# 3. 重启服务
docker-compose restart
```

### JWT 密钥泄露

```bash
# 1. 生成新的 JWT_SECRET
openssl rand -base64 64

# 2. 更新 .env 文件
# JWT_SECRET="new-secret-key"

# 3. 重启 API 服务（所有现有 Token 失效）
docker-compose restart api
```

### 恶意节点/Token

```sql
-- 撤销特定订阅 Token
UPDATE subscription_tokens SET is_revoked = 1 WHERE token = 'malicious_token';

-- 管理员重置用户 Token 时应通过后台接口完成，避免手工 SQL 生成低熵 Token。

-- 禁用特定用户
UPDATE users SET status = 'disabled' WHERE username = 'malicious_user';

-- 禁用特定节点
UPDATE nodes SET is_enabled = 0 WHERE id = <NODE_ID>;
```

## 监控指标

### 关键指标

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| API 响应时间 | > 1s | 可能数据库慢查询或负载过高 |
| MySQL 连接数 | > 80% | 可能连接泄漏 |
| node-agent 心跳间隔 | > 60s | 节点可能离线 |
| node-agent relay 模式心跳间隔 | > 60s | 中转配置可能未生效 |
| 中转监听端口可达率 | < 100% | 用户中转线路可能不可用 |
| HAProxy 配置校验失败 | > 0 | 中转配置未 reload，需检查 relay_config_tasks 错误 |
| 磁盘使用率 | > 80% | 日志或数据库文件过大 |
| 内存使用率 | > 90% | 可能内存泄漏 |

### 日志监控命令

```bash
# 监控 API 错误日志
docker-compose logs -f api | grep -i "error\|panic\|fatal"

# 监控 Worker 异常
docker-compose logs -f worker | grep -i "error\|panic"

# 监控 node-agent 心跳失败
journalctl -u node-agent -f | grep -i "error\|failed"
```
