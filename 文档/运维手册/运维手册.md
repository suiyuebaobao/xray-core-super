# 运维手册

本文档提供 suiyue（岁月）系统的日常运维、故障排查和应急响应指南。

## 系统架构概览

```
用户/客户端
    │
    ▼
Nginx (80/443) ← 静态资源 + API 反代
    │
    ├── /api/*  → 后端 API (:3000)
    ├── /sub/*  → 后端 API (:3000)
    └── /*      → 前端静态文件
                    │
                    ▼
               MySQL (容器内部，不暴露端口)

节点服务器
    │
    ├── xray-core (443) ← VLESS + Reality
    └── node-agent (systemd 服务)
              │
              ▼
         心跳 → 中心 API
         任务 ← 中心 API
         流量 → 中心 API

中转服务器
    │
    ├── HAProxy ← TCP 透传转发
    └── node-agent (AGENT_ROLE=relay)
              │
              ▼
         心跳 → 中心 API
         配置任务 ← 中心 API
         结果 → 中心 API
```

## 日常运维

### 服务启动/停止

```bash
# 启动所有服务
make up

# 停止所有服务
make down

# 仅启动 API 服务
make api

# 启动 Worker
make worker

# 启动前端开发服务器
make frontend
```

### 数据清理

```sql
-- 清理 30 天前的过期 Refresh Token
DELETE FROM refresh_tokens WHERE expires_at < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- 清理 90 天前的 EXPIRED 订单（可选，保留审计记录）
-- DELETE FROM orders WHERE status = 'EXPIRED' AND created_at < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 清理 90 天前的流量快照
DELETE FROM traffic_snapshots WHERE captured_at < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 清理 90 天前的流量账本
DELETE FROM usage_ledgers WHERE recorded_at < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 清理已完成 30 天的任务
DELETE FROM node_access_tasks WHERE status = 'DONE' AND executed_at < DATE_SUB(NOW(), INTERVAL 30 DAY);
```

## 生产前验收清单

生产发布前必须把“功能可用”和“风险边界”分开记录。当前核心路径通过 smoke 不代表零缺陷；以下检查全部通过后，才可以认为本次版本具备上线条件。

### 基础构建与测试

```bash
go test ./...
docker-compose config --services
cd frontend
npm run build
E2E_ADMIN_USERNAME=<admin> E2E_ADMIN_PASSWORD=<password> npm run test:e2e
```

Playwright 默认访问 `http://127.0.0.1:7000`，公网验收时使用：

```bash
cd frontend
E2E_BASE_URL=http://your-domain-or-ip E2E_ADMIN_USERNAME=<admin> E2E_ADMIN_PASSWORD=<password> npm run test:e2e
```

### 运行时检查

```bash
curl -i http://127.0.0.1:3000/health
curl -i http://127.0.0.1:7001/health
curl -i http://your-domain-or-ip/api/plans
systemctl is-active suiyue-api suiyue-worker
```

如果采用 Compose 运行，则改查：

```bash

docker-compose logs -f worker
```

### 订阅与节点检查

- 订阅 URL 必须返回 200，且客户端导入的是完整链接 `/sub/{token}/clash`、`/sub/{token}/base64` 或 `/sub/{token}/plain`，不是裸 Token。
- Reality 节点必须确认订阅里的 `sni/pbk/sid` 与出口节点 Xray 配置一致。
- 出口节点必须确认用户 UUID 已写入 Xray `clients`，并通过真实客户端或 Xray 本地代理完成一次出站访问。
- 多出口 IP 节点必须确认目标服务器只有一个当前角色的 node-agent，每个逻辑节点的 Xray inbound `listen` 与 outbound `sendThrough` 都绑定同一个公网 IP，订阅下发的每条线路真实出站 IP 等于对应逻辑节点 IP。
- 中转线路必须确认中转入口端口可达、中转到出口端口可达、`relay_config_tasks` 最新任务为成功状态、`relay_traffic_snapshots` 有流量记录。
- 已清理或停用的旧中转节点必须同步停用 `relays` 与 `relay_backends`，否则订阅仍可能下发不可用中转线路。
- 如果要隐藏出口 IP，必须把出口节点设置为只下发中转线路，并在出口节点防火墙只放行中转服务器 IP 访问 Xray 端口。

### 尚未覆盖的风险

- 长时间稳定性观察和并发压测。
- 支付回调验签、幂等、对账和异常补偿全链路。
- 管理后台所有新增、编辑、删除、禁用、恢复组合的破坏性测试。
- 多中转节点、多出口节点、大量用户同步任务下的 Worker 重试表现。

## 故障排查

### API 服务无法启动

```bash
# 检查端口占用
lsof -i :3000
fuser -k 3000/tcp

# 检查数据库连接
mysql -u root -p -e "SELECT 1"

# 检查环境变量
env | grep JWT_SECRET
env | grep DATABASE_URL
env | grep MIGRATE_DATABASE_URL

# 查看 API 启动日志
docker-compose logs api 2>&1 | tail -50
```

### 数据库连接失败

```bash
# 检查 MySQL 容器状态
docker-compose ps mysql

# 检查 MySQL 健康状态
docker-compose logs mysql | grep "ready for connections"

# 手动测试连接
docker-compose exec mysql mysql -u root -p -e "SELECT 1"

# 重置数据库密码（bt 面板）
bt database password <new_password>
```

### 用户反馈订阅失效

```sql
-- 检查用户订阅状态
SELECT us.id, us.user_id, us.status, us.expire_date, u.username
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE u.username = 'username';

-- 检查 Token 是否有效
SELECT token, user_id, subscription_id, is_revoked, expires_at, last_used_at
FROM subscription_tokens
WHERE token = 'token_string';

-- 检查 xray 配置中是否有该用户
-- 登录节点服务器执行：
cat /usr/local/etc/xray/config.json | grep "user_key"
```

如果订阅能导入但节点无信号，优先核对 Reality 参数：

```sql
SELECT id, name, host, port, server_name, public_key, short_id
FROM nodes
WHERE is_enabled = 1;
```

节点侧执行：

```bash
cat /usr/local/etc/xray/config.json
xray run -test -config /usr/local/etc/xray/config.json
systemctl status xray node-agent
```

`nodes.server_name` 必须等于 `realitySettings.serverNames[0]`，`nodes.public_key` 必须等于节点 PublicKey，`nodes.short_id` 必须等于 `shortIds[0]`。

如果是中转线路，继续检查：

```bash
# 检查中转入口端口
nc -vz <RELAY_HOST> <LISTEN_PORT>

# 登录中转服务器，检查到出口节点的连通性
nc -vz <EXIT_NODE_HOST> 443

# 登录出口节点，确认只放行中转 IP 时防火墙规则生效
iptables -S
ufw status
```

中转线路的 `server/port` 来自中转节点，但 `sni/pbk/sid` 必须仍然来自出口节点。隐藏出口 IP 时，还必须在出口节点防火墙限制只允许中转服务器访问 Xray 端口。

### 节点离线排查

```bash
# 1. 检查节点上 node-agent 状态
ssh root@<NODE_IP> 'systemctl status node-agent'

# 2. 检查网络连接
curl -v http://<NODE_IP>:<AGENT_PORT>/api/heartbeat

# 3. 检查 node-agent 日志
ssh root@<NODE_IP> 'journalctl -u node-agent --no-pager -n 50'

# 4. 检查中心服务数据库
SELECT id, name, is_enabled, last_heartbeat_at
FROM nodes
WHERE id = <NODE_ID>;

# 5. 重启 node-agent
ssh root@<NODE_IP> 'systemctl restart node-agent'
```

### xray-core 异常

```bash
# 检查 xray 配置
xray run -test -config /usr/local/etc/xray/config.json

# 查看 xray 日志
journalctl -u xray -f

# 重启 xray
systemctl restart xray

# 查看当前配置中的用户
cat /usr/local/etc/xray/config.json | python3 -c "
import json, sys
d = json.load(sys.stdin)
for i in d.get('inbounds', []):
    if i.get('protocol') == 'vless':
        for c in i['settings'].get('clients', []):
            print(f\"{c.get('email', 'no-email')}: {c.get('id', 'no-id')}\")
"
```

### 流量统计异常

```sql
-- 检查用户当前订阅使用量
SELECT us.id, u.username, us.traffic_limit, us.used_traffic,
       ROUND(us.used_traffic * 100.0 / NULLIF(us.traffic_limit, 0), 2) AS percent_used
FROM user_subscriptions us
JOIN users u ON u.id = us.user_id
WHERE us.status = 'ACTIVE';

-- 检查最近流量快照
SELECT node_id, xray_user_key, uplink_total, downlink_total, captured_at
FROM traffic_snapshots
ORDER BY captured_at DESC
LIMIT 10;
```

## 应急响应

### 数据库密码泄露

```bash
# 1. 修改 MySQL root 密码
docker-compose exec mysql mysql -u root -p -e "ALTER USER 'root'@'%' IDENTIFIED BY 'new_password'; FLUSH PRIVILEGES;"

# 2. 更新 DATABASE_URL 环境变量
# 编辑 .env 文件，更新 DATABASE_URL

# 3. 重启服务
docker-compose restart
```

### JWT 密钥泄露

```bash
# 1. 生成新的 JWT_SECRET
openssl rand -base64 64

# 2. 更新 .env 文件
# JWT_SECRET="new-secret-key"

# 3. 重启 API 服务（所有现有 Token 失效）
docker-compose restart api
```

### 恶意节点/Token

```sql
-- 撤销特定订阅 Token
UPDATE subscription_tokens SET is_revoked = 1 WHERE token = 'malicious_token';

-- 管理员重置用户 Token 时应通过后台接口完成，避免手工 SQL 生成低熵 Token。

-- 禁用特定用户
UPDATE users SET status = 'disabled' WHERE username = 'malicious_user';

-- 禁用特定节点
UPDATE nodes SET is_enabled = 0 WHERE id = <NODE_ID>;
```

## 监控指标

### 关键指标

| 指标 | 告警阈值 | 说明 |
|------|----------|------|
| API 响应时间 | > 1s | 可能数据库慢查询或负载过高 |
| MySQL 连接数 | > 80% | 可能连接泄漏 |
| node-agent 心跳间隔 | > 60s | 节点可能离线 |
| node-agent relay 模式心跳间隔 | > 60s | 中转配置可能未生效 |
| 中转监听端口可达率 | < 100% | 用户中转线路可能不可用 |
| HAProxy 配置校验失败 | > 0 | 中转配置未 reload，需检查 relay_config_tasks 错误 |
| 磁盘使用率 | > 80% | 日志或数据库文件过大 |
| 内存使用率 | > 90% | 可能内存泄漏 |
