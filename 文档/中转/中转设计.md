# 中转设计

## 状态

本文档定义第 9 阶段中转线路能力。当前系统已具备直连出口节点、node-agent 用户同步和订阅生成能力；中转能力作为增量扩展设计，不替换现有直连模式。

## 核心概念

| 概念 | 说明 |
|------|------|
| 出口节点 | `nodes` 表记录的真实代理出口，运行 `xray-core`，由 `node-agent` 管理用户和流量 |
| 直连线路 | 客户端直接连接出口节点 `host:port` |
| 中转节点 | `relays` 表记录的入口服务器，只做 TCP 透传 |
| 中转后端 | `relay_backends` 表记录的 `中转监听端口 -> 出口节点` 绑定 |
| node-agent relay 模式 | 部署在中转节点上的控制面运行模式，拉取配置任务并 reload 转发组件 |
| HAProxy | 第一版默认 TCP 转发组件，由 node-agent relay 模式生成配置并 reload |

## 目标

- 保留现有直连订阅和 node-agent 控制面
- 在同一个订阅 Token 内返回直连和中转多条线路
- 中转节点只做 TCP passthrough，不终止 Reality
- 支持隐藏出口 IP：订阅仅下发中转线路，并通过出口节点防火墙只放行中转 IP
- 中转配置变更具备任务、重试、状态和人工排障能力

非目标：

- 第一版不做按中转线路单独售卖
- 第一版不做中转节点用户鉴权或流量计费
- 第一版不做同一监听端口多出口负载均衡
- 第一版不适配 Nginx stream 或 gost，二者仅作为后续转发器扩展

## Agent 运行模式与 Token

中转能力复用现有 `node-agent` 工程和二进制，但运行角色必须分开：

| 模式 | 环境变量 | 绑定表 | Token | 职责 |
|------|----------|--------|-------|------|
| 出口模式 | `AGENT_ROLE=exit`、`NODE_ID`、`NODE_TOKEN` | `nodes` | 出口节点 Token | 管理 xray 用户、Reality 参数、用户级流量 |
| 中转模式 | `AGENT_ROLE=relay`、`RELAY_ID`、`RELAY_TOKEN` | `relays` | 中转节点 Token | 管理 TCP 转发端口、后端绑定和转发配置 |

用户连接中转不需要中转专用用户 Token。客户端连接时携带用户 VLESS UUID，中转节点只做 TCP 透传，出口节点 xray 最终识别该 UUID 并完成用户鉴权和用户级流量统计。

`RELAY_TOKEN` 只用于 node-agent relay 模式访问中心服务，不能用于下载订阅、拉取出口节点任务、上报用户级流量或修改出口节点 Xray clients。

中转节点一键部署必须包含后端绑定闭环。管理员在部署时选择出口节点和监听端口后，中心服务必须自动创建 `relay_backends`、写入 `RELOAD_CONFIG` 任务并等待 node-agent relay 模式回报成功，同时验证 HAProxy 监听端口已经就绪且监听进程为 `haproxy`；不能把“relay agent 心跳正常但没有后端绑定”、“reload 未确认”或“端口被其他进程监听”视为部署成功。若管理员选择替换旧角色，还必须自动停用同一物理服务器上的旧出口记录，并清理旧出口/中转 agent 容器和旧 Xray/HAProxy 配置。部署失败必须强制清理本次中转记录、后端绑定、配置任务和容器，避免留下半成品入口。

## 数据模型

### relays

中转节点表。

关键字段：`id`、`name`、`host`、`agent_base_url`、`agent_token_hash`、`agent_version`、`status`、`last_heartbeat_at`、`is_enabled`、`created_at`、`updated_at`。

### relay_backends

中转后端绑定表。一条记录就是订阅中的一条中转线路。

关键字段：`id`、`relay_id`、`exit_node_id`、`name`、`listen_port`、`target_host`、`target_port`、`is_enabled`、`sort_weight`、`created_at`、`updated_at`。

约束：

- `(relay_id, listen_port)` 唯一
- `exit_node_id` 指向出口节点 `nodes.id`
- `exit_node_id` 必须指向启用中的出口节点。保存后端绑定时拒绝停用出口；生成 `RELOAD_CONFIG` 时也必须再次过滤停用或不存在的出口，避免中转继续转发到已改作管理系统或已下线的服务器。
- `target_host` 默认取出口节点 `nodes.host`。管理后台第一版不暴露手填目标 IP，通过选择出口节点自动带出；接口字段保留给内网地址或专线地址等高级场景。
- 一个监听端口默认只绑定一个出口节点

### relay_config_tasks

中转配置任务表。

关键字段：`id`、`relay_id`、`action`、`payload`、`status`、`retry_count`、`last_error`、`scheduled_at`、`locked_at`、`lock_token`、`executed_at`、`idempotency_key`。

第一版任务动作为 `RELOAD_CONFIG`，payload 携带该中转节点完整的启用后端列表。任务由 node-agent relay 模式主动拉取，中心服务不直接 SSH 修改中转配置。

## 转发组件选型

第一版固定使用 HAProxy。

选择 HAProxy 的原因：

- 只需要四层 TCP 透传，不需要 HTTP 反代或 TLS 终止
- 配置模板简单，适合 node-agent relay 模式自动生成
- `haproxy -c` 可以在 reload 前校验配置
- 支持成熟的 reload、健康检查和连接统计
- 不会和管理平台 Web/API 使用的 Nginx 反向代理混在一起

Nginx stream 和 gost 保留为后续适配项。新增适配前必须保持同一套 `relays`、`relay_backends`、`relay_config_tasks` 抽象，不能改变订阅生成和用户计费规则。

HAProxy 配置模板示例：

```haproxy
global
    daemon
    maxconn 20000
    log stdout format raw local0
    stats socket /tmp/haproxy.sock mode 600 level admin

defaults
    mode tcp
    timeout connect 5s
    timeout client 5m
    timeout server 5m

frontend relay_1_24443
    bind *:24443
    mode tcp
    default_backend exit_1_443

backend exit_1_443
    mode tcp
    server exit1 203.0.113.10:443 check
```

node-agent relay 模式必须先把新配置写入临时文件，执行 `haproxy -c -f <临时文件>` 校验，通过后再替换正式配置并 reload。校验或 reload 失败时，必须保留上一份可用配置。

## 订阅生成

订阅生成以出口节点为授权核心：

1. 校验用户级订阅 Token
2. 查询用户当前 ACTIVE 订阅
3. 查询套餐允许的节点分组
4. 查询节点分组下启用的出口节点
5. 根据出口节点 `line_mode` 决定是否下发直连线路
6. 查询指向这些启用出口节点且启用的 `relay_backends`
7. 生成中转线路
8. 输出 Clash、Base64 或纯文本 URI

直连线路字段：

| 字段 | 来源 |
|------|------|
| server | 出口节点 `nodes.host` |
| port | 出口节点 `nodes.port` |
| uuid | 用户 xray UUID |
| sni/pbk/sid | 出口节点 Reality 参数 |

中转线路字段：

| 字段 | 来源 |
|------|------|
| server | 中转节点 `relays.host` |
| port | `relay_backends.listen_port` |
| uuid | 用户 xray UUID |
| sni/pbk/sid | 出口节点 Reality 参数 |

中转节点不是 Reality 终点，因此订阅里的中转线路不能使用中转节点自己的 Reality 参数。

## 流量统计边界

中转用户的套餐扣量仍由出口节点统计。原因是中转节点只做 TCP 透传，不解密 Reality，也不解析 VLESS UUID；但连接内容会完整透传到出口节点，出口节点 xray 仍能识别用户 UUID。

```text
客户端携带用户 UUID
        │
        ▼
中转节点 TCP 透传
        │
        ▼
出口节点 xray 识别 UUID
        │
        ▼
node-agent exit 模式上报用户级流量
```

node-agent relay 模式可以上报线路级指标：

- `relay_id`
- `relay_backend_id`
- `listen_port`
- `exit_node_id`
- 端口级上行/下行字节
- 中转到出口的连通性

node-agent relay 模式通过 HAProxy stats socket 读取 `show stat`，默认 socket 为 `/tmp/haproxy.sock`，可通过 `HAPROXY_STATS_SOCKET_PATH` 覆盖。线路级指标用于运维分析、线路质量、容量规划和风控，不直接作为用户套餐扣量依据。第一版不做中转侧用户级计费；如果未来必须在中转侧识别用户，需要引入每用户独立端口或中转终止协议，复杂度会明显增加。

## 中心地址容灾

relay 模式与出口模式共用 node-agent 的多中心机制：

- `CENTER_SERVER_URL` 保存主中心入口。
- `CENTER_SERVER_URLS` 保存主入口和备用入口列表。
- agent 访问中心失败时自动尝试下一个入口，成功后记住为 active center。

当前控制平台主域名为 `leiyunai.fun`，备用 IP 为 `154.219.106.105` 和 `154.219.106.53`。中转一键部署必须把三者写入 relay agent 容器，避免中心单个 IP 或域名不可达时中转节点失联。

如果 relay agent 已经失联，管理员通过 `/admin/relays` 的“修复中心”发起 SSH 兜底修复。中心会登录中转服务器，优先重建 `raypilot-relay-agent` / `suiyue-relay-agent` 容器并替换 `CENTER_SERVER_URLS`；如果是 systemd 部署，则写入 systemd drop-in 后重启服务。修复后可按 `relay_id` 等待新心跳确认。该兜底只修复控制面回连地址，不修改 HAProxy 后端绑定和用户订阅内容。

## 授权策略

第一版沿用现有授权链路：

```text
套餐 -> 节点分组 -> 出口节点 -> 派生直连/中转线路
```

如果套餐允许访问出口节点 A，则用户可以获得：

- A 的直连线路，前提是 A 的 `line_mode` 允许直连
- 所有指向 A 且启用的中转线路

后续如果需要“某套餐只给香港中转，不给美国中转”这类精细控制，再新增：

- `lines`
- `line_groups`
- `plan_line_groups`

不要在第一版就把套餐授权从出口节点完全改成线路授权，否则会影响当前节点组、多对多绑定和 node-agent 同步闭环。

## node-agent relay 模式控制流

```text
管理员保存中转配置
        │
        ▼
中心服务写 relays / relay_backends / relay_config_tasks
        │
        ▼
node-agent relay 模式心跳并拉取任务
        │
        ▼
渲染 HAProxy 配置
        │
        ▼
haproxy -c 校验通过后 reload
        │
        ▼
上报任务结果
```

node-agent 两种运行模式分开：

- exit 模式管出口节点用户和流量
- relay 模式管中转端口和转发配置
- 两种模式使用不同 Token
- 两种模式任务表分开，互不阻塞

## 隐藏出口 IP

隐藏出口 IP 必须同时满足：

1. 出口节点 `line_mode = relay_only`
2. 订阅中不出现出口节点直连 `host:port`
3. 出口节点防火墙只允许中转节点 IP 访问 Xray 端口
4. 中转节点监听端口对用户开放
5. 中转到出口的链路稳定可达

只隐藏订阅里的直连地址不够安全。只要出口节点 443 对公网开放，用户仍可能通过历史订阅、日志或扫描发现并直连。

## Reality 限制

一个中转监听端口默认只转发到一个出口节点。

不能把 Reality 参数不同的多个出口节点挂到同一个中转端口后面做四层负载均衡。客户端连接时会校验出口 Reality 身份，如果后端随机切到另一组 `server_name/public_key/short_id`，握手会失败。

只有在多个出口节点共享完全一致的 Reality 私钥、公钥、SNI 和 Short ID 时，才可以评估同端口多后端负载均衡。

## 验收清单

- 订阅 URL 返回 200
- 同一个 Token 返回直连和中转多条线路
- 中转线路 `server/port` 指向中转节点
- 中转线路 `sni/pbk/sid` 与出口节点 Xray 配置一致
- 中转监听端口 TCP 可达
- 中转节点到出口节点 TCP 可达
- 已停用或已改作管理系统的出口节点不会出现在中转后端选择器、`RELOAD_CONFIG` payload 或订阅中转线路里
- 用户 UUID 已写入出口节点 Xray `clients`
- 出口节点 `relay_only` 时，公网直连端口已通过防火墙限制
- node-agent relay 模式任务失败时有错误原因、重试次数和人工重试入口
- HAProxy 配置校验失败时不会覆盖上一份可用配置
